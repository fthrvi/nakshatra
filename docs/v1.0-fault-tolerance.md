## 1. The problem, stated precisely

**Today (v0.1):** if a worker dies mid-generation, the client retries the **entire request from token 0** (up to N=3 times). The dead worker's KV cache is gone; the whole conversation prefix is reprocessed from scratch on every healthy worker (`petals-architecture.md` §8.2). Transient single-worker blips are masked from the user, but at the cost of throwing away all in-flight state. For a 2-worker lab cluster this is acceptable.

**v1.0 changes the operating assumption.** Discovery (the sibling doc) lets nodes we have never configured join and leave continuously. Churn stops being an exception and becomes the steady state. Reprocessing a long prefix from token 0 every time any peer in a long chain hiccups is no longer a tolerable cost — it scales with chain length × prefix length × churn rate. **v1.0 needs in-flight recovery: when one span's worker dies, recover *only that span* and continue, without losing the tokens already produced.**

This is exactly the problem Petals solved. We adopt its mechanism and then fix the one place it breaks on our heterogeneous hardware (§4).

## 2. The reference design (Petals, accurately)

Petals' fault tolerance is three primitives in `client/inference_session.py` (`petals-deep-read.md` §6). Stated faithfully so we copy the *right* thing:

1. **A per-block retry loop, not per-request** (`inference_session.py:322-357`). The retry is *inside* the walk over blocks: if block 50's worker dies, only block 50's span is retried; blocks 0–49's outputs already live in `inputs`. `on_request_failure(peer_id)` tells the router to stop sending work to the dead peer; retries are bounded with backoff.

2. **Path re-planning over the affected range only** (`_update_sequence`, `:364-391`). On failure it asks the router for a *replacement path* covering just `[block_idx, update_end)` — never re-routing already-completed spans. The replacement may be one peer or several; `next_session` links are repaired so the server-to-server push chain stays intact across the splice.

3. **Replayable client-side history** (`:55-56`, `:113-126`). The client accumulates the hidden-state history entering each span. The replacement worker **inherits the dead worker's input history** (`:386`) and, on its first step, is sent the *full prefix* so it rebuilds the KV cache by replay — **no global checkpoint required.** The client trades memory (≈ O(t·d) hidden state stored per span boundary) for the ability to reconstruct any span.

**What Petals explicitly does *not* handle** (`petals-deep-read.md` §6.5), each a design item for us:
- No fallback below the last completed step if *every* peer dies simultaneously.
- **No correctness guarantee under non-determinism** — a replacement worker's recomputed KV must be numerically equivalent to the original's. "With deterministic FP/dtype this is fine; with non-deterministic kernels this is implicitly papered over." ← **This is the landmine for Nakshatra (§4).**
- Server-to-server `rpc_push` failures are swallowed; the client-routed path is the correctness backstop (push is pure optimization).

## 3. The Nakshatra adaptation

We already have the two hard prerequisites Petals leans on:
- **Server-to-server push** with `next_servers` metadata (`petals-architecture.md` §99) — the chain-splice repair target.
- **A peer registry with redundant slice caching** (Sthambha Phase 4 — a slice can live on several peers; the planner Q3 already specifies *partial re-plan* finding a replacement peer that holds the dead span).

So v1.0 adds, on top of v0.1's client retry:

### 3.1 Replayable per-span history (the core)
The client retains the hidden-state vector stream entering each span boundary (the input to each worker), keyed by `(layer_start, layer_end)`. This is the v0.5 "in-flight KV preservation" item from `petals-architecture.md` §6.3, made concrete. On a worker death:
1. Mark the peer failed; stop routing to it (feeds the discovery ranker's reputation, §5).
2. Ask the planner/router for a **replacement peer holding the same `[start,end)` slice** (Sthambha redundancy makes this usually a cache hit, not a re-cut).
3. Hand the replacement the retained input history for that span; it replays to rebuild KV from the span start — **not from token 0.**
4. Repair the `next_servers` push links around the splice; continue generation.

### 3.2 Bound the memory cost
Storing full per-span history is O(t·d) per boundary and grows with sequence length. Bound it: cap retained history at the configured `max_length`; for very long contexts, fall back to the v0.1 from-token-0 retry **for that span only** rather than holding unbounded state. Make the cap a config knob, not a silent OOM.

### 3.3 Hot-spare peers (planner-side)
`layer-split-planner.md` §4-step-8 already lists "keep one peer in reserve so a single failure doesn't require re-planning." v1.0 promotes this from TODO to default for chains above a length threshold: the planner provisions a warm replica of high-risk spans (the longest spans, or spans on peers with poor heartbeat history) so recovery is a *re-point*, not a *re-fetch + re-plan*.

## 4. The determinism landmine — the one place Petals' design is unsafe for us

Petals assumes a replacement worker recomputes a **numerically equivalent** KV cache. That assumption holds in a homogeneous datacenter with one kernel and one dtype. **It does not hold on Nakshatra's hardware.** Cross-vendor numerical drift is a *known, documented* property of this stack — it is the entire reason vendor-grouping exists in the planner (`layer-split-planner.md` §4 step 2, `chain_status: drifty`). An RX 9070 XT (ROCm), an Apple M-series (Metal/MoltenVK), and a CPU worker do **not** produce bit-identical activations for the same span.

**Consequence:** if span `[14,28)` was being served by a ROCm peer and we recover it onto a Metal peer, the replayed KV is *not* the KV the request was built on. The generation silently diverges — a correctness bug that looks like a quality regression, not a crash. Petals papers over this; we cannot, because heterogeneity is our headline feature.

**The rule v1.0 must enforce:** **recovery is vendor-/kernel-constrained.** A failed span may only be recovered onto a peer in the *same drift class* (same `gpu_backend`, same quant/dtype) as the peer that originally served it — or, if none is available, the request fails over to **full from-token-0 reprocessing on a fresh, internally-consistent chain** (the v0.1 behavior), never a silent mixed-kernel replay. This is a real divergence from Petals and the most important sentence in this doc:

> Petals trades memory for fault tolerance. **Nakshatra additionally trades flexibility for correctness:** in-flight recovery is only allowed within a drift class; cross-class recovery degrades to a clean restart, never a silent splice.

A second-order option (cheaper than restart, more flexible than same-class): for spans where the redundant-compute verification path (`petals-architecture.md` §"redundant compute") is already running, an ε-tolerance band can permit cross-class recovery if the replacement's first verified token matches within tolerance — but default to same-class, treat ε-cross-class as opt-in.

## 5. Interaction with discovery, trust, and the planner

- **Reputation feedback.** Every `on_request_failure(peer_id)` is signal. Wire it back into the discovery ranker (`v1.0-discovery-and-distribution.md` §4.3): peers that drop mid-generation lose rank, so the mesh organically routes around flaky strangers. This is the missing reputation layer that signed listings alone don't provide.
- **Trust (`THREAT_MODEL.md`).** Recovery resends real hidden-state history to a *replacement* peer. That peer must clear the same admission wall (token + Ed25519 pin) as any other — recovery is **not** a bypass. History replay must be idempotent and replay-protected (request-ID dedup, already flagged SOLVING in `research-findings-topology-resilience.md`) so a malicious or buggy replacement can't double-execute or forge progress.
- **Planner.** Recovery reuses the planner's *partial re-plan* (Q3) and *redundant slice cache* (Phase 4); it does not invent a new placement path. The planner just gains a "replace this one span, same drift class" entry point.

## 6. Phasing

| Phase | Capability | Builds on |
|---|---|---|
| **v0.1 (shipped)** | Client full-request retry from token 0, N attempts | — |
| **v0.5** | §3.1 replayable per-span history + same-span re-point on a *known* cluster (no discovery yet); §3.2 memory bound | petals-architecture §6.3 deferral |
| **v1.0** | §3.3 hot-spare provisioning; §4 drift-class-constrained recovery; §5 reputation feedback into the discovery ranker; replay-protected admission of replacement peers | this doc + discovery sibling |
| **later** | §4 ε-tolerance cross-class recovery gated on redundant-compute verification; multi-failure cascade handling (Petals §6.5 gap) | redundant-compute verification path |

## 7. Acceptance test (the v1.0 FT ship gate)

On a ≥3-worker chain serving a long-prefix request:
1. **Kill one mid-span worker at token T (T ≫ 0).** Generation continues; the user sees a brief stall, not a restart. The replacement serves the *same* `[start,end)` from a same-drift-class peer; the **final output is byte-identical** to the no-failure run (the parity bar, now under failure).
2. **Kill a worker when only a *different-vendor* replacement is available.** The request must either (a) recover onto a same-class peer if one exists, or (b) cleanly restart on a consistent chain — and must **never** splice a mixed-kernel KV and silently diverge. Assert the output is still correct (matches a single-machine reference), not merely "didn't crash."
3. **Churn storm:** peers join/leave repeatedly during one generation. The request completes; failed peers lose discovery rank; no unbounded memory growth (the §3.2 cap holds).
4. **A replacement peer presenting an invalid token/pubkey is rejected** — recovery is not an admission bypass.

If all four hold, Nakshatra survives a mesh of churning strangers without sacrificing the correctness its heterogeneity would otherwise threaten.

## 8. Open risks
- **Memory vs context length.** Per-span history storage caps how long a context we can recover in-flight; §3.2's fallback is a real degradation, not a free lunch — measure where it bites.
- **Same-drift-class availability.** In a small or vendor-skewed mesh there may be *no* same-class replacement → frequent restarts. Discovery ranking should prefer growing same-class redundancy for live chains.
- **Cascade failure** (Petals §6.5): simultaneous death of every peer covering a span range still falls back to from-token-0. Multi-failure recovery is deferred.
- **Determinism drift within a vendor.** Same `gpu_backend` is necessary but maybe not sufficient (driver/kernel version skew). The drift class may need to key on (backend, quant, kernel-version) — validate empirically before trusting same-vendor as same-class.
