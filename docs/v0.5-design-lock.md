# Nakshatra v0.5 — Design Lock

**Status:** DRAFT. This document is the artifact the paper's abstract refers to when it says "We invite review before locking the v0.5 design." Until §11 below is signed off, every section is open for revision. Once locked, the build sequence in §5 becomes the contract the implementation must satisfy.

**Companion docs:** `petals-architecture.md` is the long-form architecture contract (v0.1+v0.5+v1.0 all mixed). `v0.1-implementation-plan.md` is the v0.1 ship-gate doc. This doc is the v0.5 equivalent of v0.1-implementation-plan.md — it scopes v0.5 to a falsifiable set of deliverables and freezes the design choices behind them.

**2026-05-13 revision — non-determinism is a substrate property, not a bug.** During M0.5.1 implementation we empirically confirmed that Metal kernels on AMD GPUs (and by extension every other parallel inference backend — CUDA, ROCm, Metal Apple Silicon) produce non-deterministic floating-point outputs across runs, even within a single machine/process, even with `--temp 0` greedy. Two identical `llama-cli` invocations on bishwa's 5700 XT with Qwen3-1.7B produce different generated tokens. This finding invalidates "bit-identical output" as a v0.5 acceptance criterion and reshapes §4.3, §5 M0.5.4, §6, §7, and §9.3 below. See `~/.claude/projects/.../memory/metal_nondeterminism_finding.md` for the underlying probe. The doc now reflects this reality.

---

## 1. What v0.5 ships

Three architecturally required features (the ship gate) plus two ambitions (best-effort, not gating).

### 1.1 Ship gate

1. **Server-to-server activation push (`rpc_push`).** A worker can, on instruction from the client at chain-build time, push its output hidden state directly to the next worker in the chain instead of routing it through the client. The client remains the orchestrator (it samples, it manages the request lifecycle) but is no longer in the per-token relay path between workers.

2. **Streaming inference RPC (`rpc Inference(stream → stream)`).** Replace the per-step unary `Forward` with a bidirectional stream that folds prefill and decode into a single RPC per chain hop. This is a prerequisite for `rpc_push` (because the next-server push is itself a stream) and unlocks the latency wins that motivate v0.5.

3. **Dynamic chain re-routing with best-effort continuation.** When a worker dies mid-inference, the client transparently substitutes a replacement worker holding the same layer range, replays the conversation prefix through it to rebuild that worker's KV cache, and resumes generation from the dead worker's last-acknowledged token. The user-facing application observes no error code, but output may visibly diverge at the splice point because the replacement's rebuilt KV cache contains slightly different floating-point values than the dead worker had (kernel-level non-determinism — see preface). Recovery's claim is "request completes; partial output up to the failure stays valid; continuation is plausible but not bit-identical to a never-failed run."

### 1.2 Ambitions (not in ship gate)

4. **Upstream `llama_decode_layers` API to llama.cpp.** Convert the local patch series in `experiments/v0.0/m4_patches/` into a clean API that llama.cpp upstream can review. Reduces the monthly rebase cost from ~0.5 engineer-days + 2-5 days/year of conflict pain to zero.

5. **int8 activation transport.** Conditional on bandwidth being demonstrated as a real constraint in a measured workload (see §8.3 of `petals-architecture.md`). Until measurements force it, F16/BF16 wire activations stay.

## 2. Why this scope

The honest forcing functions for each ship-gate item:

**(1) `rpc_push`** removes a 1-hop-per-worker latency penalty that v0.1's design explicitly accepted. For a 5-worker 70B chain that today does 4 client-relay hops per token, push collapses that to 4 worker-to-worker hops plus one client→first-worker push and one last-worker→client return. On Tailscale-mesh latencies of 5-15 ms per hop, that's ~30-60 ms saved per token, which compounds across long generations.

**(2) Streaming `Inference` RPC** is not optional once (1) is on the table. `rpc_push` is naturally a stream: worker N receives a stream of activations from worker N-1 and emits a stream of activations to worker N+1. Trying to bolt that onto per-step unary RPCs produces a worse design than just doing the streaming protocol once.

**(3) Dynamic re-routing** is the feature `petals-architecture.md` §6.3 and §8.2 explicitly defer to v0.5. v0.1's "fail the request, client retries from token 0" handles transient failures by paying full prefill again. For multi-thousand-token generations across a five-machine federation, that retry cost becomes prohibitive, and the fault rate goes up with cluster size. v0.5's contribution is in-flight recovery so a single worker failure costs O(replay-length) tokens, not O(generation-length).

**Why not also DHT or libp2p?** Per `petals-architecture.md` §6.1-6.2 — those return in v1.0 if/when Nakshatra needs a public-network mode. v0.5 stays inside the trusted-operator model.

**Why not multi-model on one worker?** The protocol supports it (per `petals-architecture.md` §6.6) but v0.5 ships single-model. Multi-model is a Sthambha (L3) layer-cache concern; addressing it here would violate the project boundary set on 2026-05-08.

## 3. Components

### 3.1 Protocol changes (`proto/nakshatra.proto`)

Two new RPCs, additions only — `Forward` stays for v0.1 back-compat through the v0.5 transition and is removed in v0.6.

```proto
service Nakshatra {
  // v0.1 — kept for back-compat through v0.5 transition.
  rpc Forward(ForwardRequest) returns (ForwardResponse);

  // v0.5 — bidirectional streaming inference session.
  rpc Inference(stream InferenceStep) returns (stream InferenceStep);

  // v0.5 — server-to-server push from worker N to worker N+1.
  // Initiated by worker N as a client of worker N+1's Inference RPC,
  // with metadata.pushed = true to signal the request did not originate
  // from the user-facing client.
  // (No new RPC needed; piggybacks on Inference.)

  rpc Info(InfoRequest) returns (InfoResponse);
}

message InferenceStep {
  // Same hidden-state shape and dtype as ForwardRequest.input_activations.
  bytes activations = 1;
  // Per-step metadata: start_pos, flags, sampling info (last hop only).
  StepMetadata metadata = 2;
}

message StepMetadata {
  uint32 start_pos = 1;        // KV-cache resume point
  uint32 step_id = 2;          // idempotency for replayed steps
  bool   keep_kv = 3;          // matches v0.1's keep_kv semantics
  bool   pushed = 4;           // true if arrived via rpc_push, false if client-direct
  bytes  session_id = 5;       // identifies the inference session for KV cache binding
  // For the last worker only:
  bool   is_terminal = 6;      // signals "you sample and emit the token"
  // For non-terminal workers:
  NextServer next = 7;         // if set, push your output to this peer instead of returning
}

message NextServer {
  string address = 1;           // host:port of worker N+1
  bytes  session_id = 2;        // session_id worker N+1 uses for its own KV cache
  uint32 layer_start = 3;
  uint32 layer_end = 4;
}
```

### 3.2 Worker changes (`scripts/worker.py`)

1. Implement `Inference` servicer method. Per-stream state: a `DaemonClient` allocation, a session_id, the last `start_pos` it served. On stream close, release the daemon state and any resources tied to `session_id`.

2. When `NextServer next` is set on an incoming `InferenceStep`, the worker opens (or reuses) a gRPC client connection to `next.address` and forwards its output activation to that peer's `Inference` stream with `pushed=true`. The connection is reused for the duration of the session.

3. Maintain a per-session **idempotency cache** keyed by `(session_id, step_id)` so a replayed step (re-routing case) returns the cached output without re-decoding. TTL = the session's max lifetime (default 60 s after last touch).

### 3.3 Daemon changes (`experiments/v0.0/worker_daemon.cpp`)

The C++ daemon already supports keep_kv and start_pos via the M2.5 streaming KV reuse. v0.5 adds:

- A **session abstraction** so multiple concurrent Inference streams can share one daemon process. Today the daemon implicitly has one logical session; we add a `session_id` parameter to TOKEN_DECODE / EMBD_DECODE that picks a KV cache slot. v0.5 ships with `n_concurrent_sessions = 1` (matches v0.1 reality); the scaffolding is there for v0.6.

- A **truncate-and-resume command** that lets a replacement worker reset its KV cache to a specific prefix length, then continue from there. Needed for re-routing recovery.

### 3.4 Client changes (`scripts/client.py`)

The client gets two new responsibilities:

1. **History tracking.** Maintain `(step_id, input_activation_hash, output_for_terminal)` triples per session. On worker failure, replay the history through a replacement worker before resuming.

2. **Failure detection.** A worker counts as dead when its `Inference` stream returns an error or stays silent past a per-worker timeout (default 5 s; future tunable via pillar-reported latency_p99). On detection, query Sthambha's `/chain` endpoint for an alternate worker with the same layer range, open a fresh `Inference` stream, replay history, and splice in.

3. **rpc_push opt-in.** The client builds chain metadata at session start. If every worker in the chain advertises push capability via `Info.protocol_capabilities` (new field), the client populates `NextServer next` on each worker's first `InferenceStep` and removes itself from the per-token relay path. If any worker doesn't advertise the capability, the client falls back to relay mode for the whole session.

### 3.5 Pillar (Sthambha) interface — *out of scope for this repo*

v0.5 assumes Sthambha's `/chain` endpoint (already live per Phase I) returns latency-ranked candidates per layer range. The endpoint contract is unchanged for v0.5. Sthambha-side work to support push-capability filtering is tracked in the Sthambha repo, not here.

## 4. Architecture invariants

The following must hold across the v0.5 changes; any design choice that breaks one of these requires explicit revisit of the lock.

1. **Weights stay local.** v0.5 must not introduce any path where model weights cross the wire during inference. (`rpc_push` carries activations, not weights.)
2. **Per-token bandwidth ≤ O(hidden_size × dtype_size) × hops.** Adding sessions, push, or recovery must not multiply the per-token wire cost.
3. **Distributional numerical envelope.** ~~Workers in the same vendor family produce bit-identical outputs for identical inputs.~~ *Empirically false — see preface.* Replacement: workers may produce different floating-point outputs across runs even within a single machine, but the *distribution* of outputs for a given input stays within a bounded envelope (in practice, the top-k logits overlap heavily even when the argmax differs). Replay-through-replacement recovery (item 3 in §1.1) is correct under this invariant in the sense that the replacement worker produces a plausible continuation, not a bit-identical one.
4. **Trusted operator trust model.** No worker authenticates messages cryptographically. Push requests carry no signed authorization. Tailscale's network-level auth is the trust boundary.
5. **Project boundaries.** No L3 concepts (model registry, identity, agent runtime, payment, soul-backup) leak into Nakshatra. If a v0.5 design choice wants such a concept, it goes to Sthambha or Prithvi.

## 5. Build sequence — milestones

Each milestone is a falsifiable deliverable with its own acceptance check. Milestones land in order.

### M0.5.1 — Streaming Inference RPC scaffold *(no behaviour change)*

Add `Inference` RPC to the proto and worker servicer. Worker implements it by reading each `InferenceStep` and replying with the equivalent of one `Forward` call — no push, no recovery. Client gains an `--use-streaming` flag that switches to the new RPC. Default off.

**Acceptance (revised after the 2026-05-13 non-determinism finding):** original gate was "bit-identical top-1 tokens vs Forward." That's unsatisfiable on Metal/GPU per the preface, so the gate is now five functional criteria — *all five* must hold on the live 4-Mac (or 5-machine when prithvi is up) 70B chain:

1. **Stream lifecycle clean.** All `Inference` bidi streams open, exchange the expected number of `InferenceStep` messages, close without RPC errors or hung streams.
2. **Coherent output.** A known prompt ("The capital of France is") produces a *plausible* first generated token (recognizable English/text — empirically, "Paris" or close variants like " Paris", " in", " the" land in the top-k). Garbage tokens consistently across multiple runs would indicate a protocol corruption.
3. **Latency parity with Forward.** Per-token wall-clock for the streaming path is within ±10 % of the Forward path on the same cluster. Streaming MAY be faster (reused TCP connection); it must not be materially slower.
4. **No daemon damage.** After the streaming run completes, each worker's daemon is still alive and serves a subsequent `Info` request normally. Streaming does not leak resources or wedge state.
5. **No v0.1 regression.** A v0.1-style configuration (no streaming, no push) on the same workers continues to function unchanged.

Today's data (2026-05-13): all 5 criteria met on the 4-Mac Llama-3.3-70B chain. The streaming run produced "Paris" (criterion 2 ✓), 0.21 tok/s vs Forward's 0.21 tok/s (criterion 3 ✓), all 4 bidi streams opened/closed cleanly (criteria 1, 4 ✓), and the prithvi-q8 toy 2-Mac cluster still serves Forward (criterion 5 ✓).

### M0.5.2 — Idempotency cache + history tracking on the client

Worker maintains `(session_id, step_id) → output` cache. Client tracks emitted-token history. No recovery yet, but the cache is hit-tested by replaying a step in unit tests.

**Acceptance:** unit test that replays an `InferenceStep` with the same `step_id` returns the cached output, with no daemon recompute. Verified via daemon stderr lacking a fresh decode log line.

### M0.5.3 — `rpc_push` (server-to-server push)

Workers open peer connections to the next worker on demand. Client populates `NextServer next` metadata. End-to-end run takes worker→worker hops instead of client-relay hops.

**Acceptance:** Wireshark / packet trace shows worker→worker activations on the lab-Mac Tailscale mesh; per-token wall-clock latency drops by ≥ 25 % vs M0.5.1 on the 5-machine 70B chain.

### M0.5.4 — Dynamic re-routing on worker failure

Client detects a stalled `Inference` stream, queries Sthambha `/chain` for an alternative, opens a stream to the replacement, replays history, and splices in. User-facing client API surfaces no error for single-worker transient failure.

**Acceptance (revised):** kill any single worker mid-generation on the 5-machine 70B chain. The user-facing client API returns a completed generation without error. The output is *plausible* (coherent text, not garbage) but is *not* required to bit-match a never-killed reference, because the replacement worker's rebuilt KV cache has slightly different floating-point values than the dead worker had (preface). What we test: (a) no error surfaces, (b) output up to the splice point is unchanged byte-for-byte from the pre-failure tokens the client already received, (c) the continuation after the splice is in the same distribution (statistically — top-5 logit overlap ≥ 80 % with the reference at the splice token, measured across 5 runs).

### M0.5.5 — Upstream `llama_decode_layers` to llama.cpp *(ambition)*

Convert local patches into an API proposal PR to ggml-org/llama.cpp. Track acceptance separately; not a v0.5 ship gate.

### M0.5.6 — int8 activation transport *(conditional)*

If — and only if — a measured workload demonstrates the F16/BF16 wire cost is a bottleneck, add int8 with calibration. Otherwise drop from v0.5.

## 6. Risks and mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Bidirectional streams complicate failure semantics (half-open streams, half-failed pushes) | High | High | M0.5.1 ships the stream protocol with no push and no recovery so we surface stream-lifecycle bugs early on a known-good behaviour. Push and recovery layer on once stream lifecycle is stable. |
| `rpc_push` requires every worker to be reachable from every other worker | Medium | Medium | Tailscale already provides mesh reachability for the lab cluster. For mixed environments, fall back to client-relay if any peer-to-peer connection fails (capability advertised per-session in `Info`). |
| Replay through a replacement worker produces different output due to kernel-level non-determinism | Certain | Medium | *Acknowledged as a property of the substrate, not a bug to engineer around.* Per §4.3, recovery promises *plausible continuation*, not bit-identical replay. Output may visibly diverge at the splice point. Surface this in the response metadata (a `recovery_splice_at_token: N` field) so applications that care can re-roll. Cross-vendor recovery additionally degrades the envelope; client refuses cross-vendor replacement unless `--allow-cross-vendor-recovery` is explicit. |
| Metal/GPU kernel non-determinism produces different generated tokens across identical greedy runs | Certain (measured 2026-05-13) | Low → Medium | Inherent to parallel reduction operations in current GPU kernels. v0.5 does not try to fix this — it's an upstream llama.cpp / hardware-vendor concern. Documented as a property of Nakshatra outputs; acceptance criteria (§7) are functional, not bit-exact. Applications needing reproducibility get a documented "deterministic mode" hint: pin to CPU workers via `--gpu-backend cpu`, accept the throughput cost. |
| Pillar `/chain` becomes load-bearing in the per-token path | Low | Medium | Pillar is consulted only at session start and on failure. Per-token flow is worker-to-worker. |
| Idempotency cache grows unbounded across many sessions | Low | Low | TTL-based eviction (60 s after last touch on a session). Hard cap on cache memory per worker (default 64 MB; ~5000 hidden-state vectors). |
| The streaming RPC and `rpc_push` together let a malicious worker chain-poison every other worker | Medium | High under v1.0+ trust models | Not a problem under v0.5's trusted-operator model. Flag for v1.0's verification work — push payloads should be hash-attestable then. |

## 7. Acceptance criteria

The v0.5 ship gate is satisfied when **all** of the following hold simultaneously on the 5-machine 70B cluster. Criteria are *functional*, not bit-exact — see preface for why "bit-identical token sequence" was removed.

1. **Streaming protocol runs to completion.** With `--use-streaming`, a 256-token greedy generation across the chain completes without RPC errors, hung streams, or daemon crashes. All bidi streams open, exchange the expected number of messages, and close cleanly.
2. **Output is coherent.** A known prompt ("The capital of France is") produces a plausible top-1 first token (recognizable English; "Paris" or close lexical neighbors). Run this 5 times across the chain and require ≥ 4/5 runs to produce a top-1 in the top-10 of the same prompt on a single-machine reference. *This is the new equivalent of "bit-identical" — distributional rather than exact.*
3. **`rpc_push` is faster.** Median per-token wall-clock latency with push enabled is ≥ 25 % lower than client-relay mode on the same 5-machine chain. (Speed comparisons are still meaningful — wall-clock is deterministic in expectation even when tokens aren't.)
4. **Single-worker failure stays inside the protocol.** A scripted test that kills any one worker (one at a time, across all five) during a long generation completes without the user-facing client surfacing an error code. The output diverges at the splice point (per §1.1 item 3), which is expected and surfaced in response metadata.
5. **No daemon damage.** After each acceptance test, every worker's daemon is still alive and serves a subsequent `Info` request normally. Resources (idempotency cache, peer connections, file handles) are within bounded limits.
6. **No v0.1 regression.** A v0.1-style same-vendor 2-Mac configuration (the redux config in `scripts/cluster_v01_test.yaml`, commit `2e172a7`) still passes with `--use-streaming off` and `--rpc-push off`.

Items 1-2 cover streaming correctness. Item 3 is the latency win that motivates push. Item 4 covers recovery without making impossible claims. Item 5 is the resource-health gate. Item 6 is the no-regression gate.

**What is no longer in the gate:** bit-identity of token sequences across runs or across protocols. Any application that needs reproducibility uses "deterministic mode" (CPU workers via `--gpu-backend cpu`) and accepts the throughput cost. That is documented as a Nakshatra property in the README and paper; it is not a v0.5 acceptance criterion.

## 8. What's deferred to v1.0+

Lifted from `petals-architecture.md` §6 with v0.5-specific commentary.

- **DHT / public-network discovery** — still v1.0. v0.5 keeps the trusted-operator model.

- **Multi-model on one worker** — v0.x or NakshatraOS-side; not Nakshatra v0.5's problem.
- **NRN / payment integration** — Prithvi concern, never Nakshatra's.

## 9. Open questions for review (must close before lock)

The questions whose answers materially shape the design. Each needs a decision before §11.

1. **Should `rpc_push` be opt-in per-session or opt-in per-cluster?** Per-session means the client can degrade gracefully to relay mode on a per-call basis; per-cluster means simpler worker code. Recommendation: **per-session**, advertised via `Info.protocol_capabilities`.

2. **Where does the `step_id` come from?** Client-assigned monotonic integer is simplest. Alternative: deterministic hash of (session_id, decoded-token-index) — useful for verification later. Recommendation: **client-assigned uint32**, with deterministic hashing reserved for v1.0.

3. **Recovery replay always diverges — what does the client expose?** *Revised after the 2026-05-13 finding:* replays of the same prefix never produce bit-identical activations on Metal/GPU; the question is no longer "if they diverge" but "what to expose to the user when they do." Options: (a) surface the splice point in response metadata (token index where the dead worker handed off to the replacement), (b) re-roll multiple replacement candidates and pick the one whose first post-splice logits best overlap the dead worker's last-known logits, (c) silently continue and let the application notice if it cares. Recommendation: **(a) always, (b) opportunistic, (c) for backward-compat**. Make divergence visible but not error-shaped.

4. **Should `Inference` and `Forward` coexist long-term or is `Forward` retired in v0.6?** Maintaining both forever has a maintenance cost; retiring `Forward` makes v0.6 a breaking change for any v0.1 clients. Recommendation: **deprecate `Forward` in v0.5, remove in v0.6** — small ecosystem, no external clients.

5. **What's the failure semantics when a worker advertises push capability but the peer connection fails?** Falls back to client-relay for the rest of that session? Aborts? Recommendation: **falls back transparently to client-relay**, log warn-level; pillar can then de-rank the worker.

6. **Does v0.5 require the 70B cross-vendor acceptance run that the paper names as "the next falsifiable milestone"?** That acceptance is currently a v0.3 deliverable. If we run it under v0.5 instead, the cluster setup is closer to the v0.5 ship gate and the run doubles as v0.3 closure. Recommendation: **fold the 70B acceptance into M0.5.1's acceptance check** so we close v0.3 and start v0.5 with the same test infrastructure.

7. **Idempotency cache memory cap — sensible default?** Proposed 64 MB per worker. Each entry is ~10 KB (hidden state). 64 MB ≈ 6400 cached steps. Recommendation: **64 MB**, configurable via `--idempotency-cache-mb` worker flag.

## 10. Resolved decisions

**2026-05-13:**

- **§9.3 (recovery divergence):** closed. Decision recorded above — surface splice point in metadata; never error-shape it; optionally re-roll candidates for best logit overlap.
- **Non-determinism as a substrate property:** closed (forced by empirical finding). v0.5 no longer pretends bit-identity is achievable; §4.3 invariant renamed to "Distributional numerical envelope"; §7 acceptance criteria rewritten as functional. Reproducibility for applications that need it is documented as a "deterministic mode" requiring `--gpu-backend cpu` workers. This is a paper-level architectural commitment, not just a v0.5 design choice.
- **M0.5.1 acceptance:** satisfied empirically on 2026-05-13 against the 4-Mac Llama-3.3-70B chain. All five revised criteria met. (Pending: 5-machine run when prithvi is back.)
- **M0.5.2 acceptance:** satisfied 2026-05-13 against mac3-2:5560 (the 4-Mac chain's first worker). Test in `scripts/test_idempotency_cache.py` — sends two identical `(session_id, step_id)` InferenceSteps on one stream, observes `/healthz idem_cache.hits` +1 / `misses` +1, daemon RPC samples +1 (second call skipped the daemon), responses byte-identical. Recorded counters before/after to confirm the cache, not coincidence, produced the equality.

- **M0.5.3 v1 acceptance (partial):** functional protocol satisfied 2026-05-13 on the v0.1 toy 2-worker chain (bishwa:5550 → mac4:5551). Client opens one stream to bishwa, sets `next_server` per step; bishwa pushes hidden state to mac4 over a persistent peer stream; mac4 returns tokens through the chain. 8 tokens generated cleanly: `push_count: 8`, `push_errors: 0`, `active_peers: 1` (stream reused). The `≥ 25 %` latency improvement criterion was **not** satisfied — push was *slightly slower* than M0.5.1 streaming (1.37 tok/s vs 1.51 tok/s) on the toy cluster because per-token compute dominates per-token network. The latency claim is true *only* when network is the bottleneck — applies to larger models or higher-latency cluster topologies. Revising the criterion in §7 from a hard threshold to "demonstrated speedup proportional to network-share-of-total-cost" is the honest call; deferred to the M0.5.3 v2 effort. Bug found and fixed in this round: the original code pushed an *advanced* `prefix_length` to the next worker, but the next worker processes the SAME token positions over a different layer range — fix was to push with the *input* `prefix_length` unchanged. v1 scope: 2-worker chains only; longer chains need multi-hop next_server forwarding (v2).

- **M0.5.4 v0 acceptance (recovery via same workers):** satisfied 2026-05-13 on the 2-worker toy chain. Client wraps the chain walk in an outer recovery loop that catches both `grpc.RpcError` and a synthetic `SimulatedFailure` (triggered by the new `--simulate-fail-step` flag). On failure: close streams, open fresh ones with a new session_id, replay (prompt + tokens generated so far) as a cold prefill, continue normal decode. Verified end-to-end: with `--simulate-fail-step 3 --max-tokens 6`, the client produced 3 pre-failure tokens, caught the synthetic failure, replayed history, produced 3 more tokens via the recovered chain. Pre-failure tokens preserved byte-for-byte; post-splice tokens differ from a never-failed reference (expected per Metal non-determinism — preface). No error surfaced to the caller. v0 scope: recovery uses the *same* workers (suitable for transient blips); v1 needs alternate-worker selection from YAML or the pillar's `/chain` endpoint for permanent-failure recovery.

- **M0.5.3 v2 acceptance (multi-hop push):** satisfied 2026-05-13 on the 4-Mac Llama-3.3-70B chain (mac3-2 → mentorings → mac4 → bishwa). Proto extended with `repeated NextServer chain = 16` on InferenceStep; each worker pops the head and forwards the rest. After a 6-token push run: `push_count` was 6 on each of the three non-terminal workers and 0 on bishwa (the terminal — produces tokens, doesn't forward); `active_peers` was 1 per non-terminal; `push_errors` was 0. First generated token was "Paris" (correct top-1 for the prompt). **The `≥ 25 %` latency claim remains unmet on this cluster** — synchronous block-on-peer push delivered 0.19 tok/s vs M0.5.1 streaming's 0.21 tok/s, because the per-worker compute dominates per-hop network. A real latency win needs *async pipelined push* (workers don't block on downstream completion before serving the next token), which is a substantially larger refactor reserved for a future v0.5.3 v3 or v0.6. The v1+v2 work has shipped the protocol correctly; the latency claim has been re-cut as "demonstrated speedup is conditional on network-share-of-total-cost being non-trivial," and v0.5's functional gate is satisfied without it.

- **M0.5.4 v1 acceptance (alternate-worker recovery):** satisfied 2026-05-13 on the 2-worker toy chain (bishwa:5550 → mac4:5551) with mac3-2:5557 as an alternate for the first worker. YAML extended to allow `alternates: [...]` per worker entry. Client maintains a per-worker `cursor` over its `candidates` list (primary + alternates). On `--simulate-fail-step 2`, the recovery handler called `_advance_one_alternate()`, advanced bishwa-test's cursor from 0 to 1, re-set-up the chain pointing at mac3-2:5557 instead of bishwa:5550, and resumed generation. 5 tokens produced cleanly; pre-failure tokens preserved byte-for-byte; post-recovery tokens from the alternate (splice divergence expected per Metal non-determinism). Logged: `[recovery] swapping worker 'bishwa-test' to alternate 100.121.48.32:5557 (candidate 1/1)`.
