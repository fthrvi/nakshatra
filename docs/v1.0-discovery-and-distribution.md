## 1. The one-sentence problem

Nakshatra today is a **hand-wired chain**: a static YAML lists each worker's address and layer range, and reachability is Tailscale-only. That ceiling is the single thing standing between "a cluster I assembled by hand" and "a network strangers can join." This doc specifies how to break it **without touching the compute-bound data path** and **without weakening the signed, closed mesh.**

## 2. Why this is the right next structural move

The "Compute, Not the Wire" thesis (Zenodo DOI 10.5281/zenodo.20514966) says the per-token wire cost is trivial (~16 KB/token for a 70B); the binding constraint is per-node compute. **Discovery and weight distribution are both control-plane / staging concerns — they cost nothing per token.** So they are free to improve under our own thesis: we can make the mesh self-assembling and self-provisioning without spending a single byte on the inference hot path.

The independent confirmation is **Mesh-LLM** (github.com/Mesh-LLM/mesh-llm — Rust, iroh-QUIC, vendored llama.cpp, Apache-2.0, very active). It made the *same* core bet as us — pipeline-split by contiguous layer ranges, weights local, only activations on the wire — and then built exactly the two pieces we lack: **automatic discovery (Nostr)** and **content-addressed weight distribution (layer packages)**. It is a networking achievement on top of stock llama.cpp. Its weaknesses are precisely our strengths: it has **no published throughput numbers** and a **public mesh where relay/worker nodes see plaintext prompts** (QUIC transport encryption only, no end-to-end). We adopt its *patterns*, not its *trust model*.

**Design rule for all of v1.0: borrow the mechanism, keep our wall.** Nakshatra's data plane stays Ed25519-signed, SPKI/TLS-pinned, and admission-gated. Discovery is allowed to be public gossip; *joining and serving* are not.

## 3. Scope

**In scope (v1.0):**
1. **Decentralized discovery** — peers find each other without a hand-edited YAML (§4).
2. **Layer-package weight distribution** — content-addressed, manifest-verified, fetch-only-your-stage (§5).
3. **A routing entry-proxy** — any node accepts an OpenAI-compatible request and routes it (§6).
4. **Wire versioning** — version-negotiated control plane so mixed-version strangers interoperate (§7).
5. **Opt-in verification hooks** — the seam for later proof-of-work receipts, not the receipts themselves (§8).

**Explicitly out of scope (deferred):**
- Mixture-of-Agents / multi-model ensembling — orthogonal to our single-stream 70B job; spends the scarce resource (compute) and re-opens the prompt-exposure surface. (§9, and report §6 P5.)
- Fault-tolerant per-token recovery (Petals O(t) dual-cache) — its own work item; discovery makes it *necessary* but doesn't deliver it.
- Batching — the paper classifies it "low value, link is not the bottleneck"; keep it out under single-request load.
- MoE-across-machines correctness — a numerical bug (FP drift flipping top-k gating), tracked separately; v1.0 is Llama-dense-only.

The acceptance test (§10) is: **two machines that have never been configured to know about each other discover, mutually authenticate, provision their assigned layer stages from a package, and produce the byte-identical greedy token the single-machine reference produces** — the v0.1 parity bar (token id 12366, `' Paris'`), now with zero hand-wiring.

## 4. Decentralized discovery (P1 — the highest-leverage change)

### 4.1 What Mesh-LLM does
A node publishes a `MeshListing` as a **Nostr Kind-31990 event** carrying `invite_token`, `serving`/`wanted` model IDs, `total_vram_bytes`, `node_count`, `mesh_id`. Peers discover with `serve --auto` / `--discover <name>` and rank with a `score_mesh()` function; the data plane joins via **iroh QUIC with relay NAT traversal**, using the `invite_token` from the listing as the join credential. Private meshes skip Nostr entirely (`serve --join <token>`).

Two corrections from the code read (don't repeat the marketing version): the event **reuses kind-number 31990 but is *not* NIP-89-conformant**, and `score_mesh` **does not rank by freshness** — its real factors are name-match (+300/−200), sticky `mesh_id` (+500), client-capacity penalty (−1000 when full), `node_count` (×5), and model match (serving ×10 / wanted ×15). The trust model is thin: listings are effectively unsigned and can lie about VRAM/serving.

### 4.2 What Nakshatra adopts — *the separation of concerns, not the trust model*
The reusable architecture is the **three-way decoupling**: **discovery (Nostr gossip) ≠ admission (a credential) ≠ transport (the pinned data plane).** Keep all three separate.

- **Discovery layer (public):** publish a `NakshatraListing` to a Nostr relay (self-hostable; Mesh-LLM ships a Fly.io relay as proof). Carry: `serving`/`wanted` model IDs, `total_vram_bytes`, `node_count`, `mesh_id`, **and — the critical addition — the node's Ed25519 public key.**
- **Admission layer (gated):** the listing's pubkey is what the join handshake **pins against**. A peer that discovers us still cannot join unless it presents a valid token *and* the handshake verifies against the advertised Ed25519 key. This closes the exact gap Mesh-LLM leaves open (unsigned listings, open join).
- **Transport layer (unchanged):** the data plane stays Tailscale + SPKI/TLS-pinned + Ed25519-signed RPC, per `THREAT_MODEL.md`. Discovery tells you *who exists*; it never widens *what they can do*.

### 4.3 Ranking — drive it off measured compute
Borrow `score_mesh`'s useful factors (sticky `mesh_id` for session continuity, capacity penalty, larger-mesh preference) but **rank candidate peers by measured `Fᵢ`** (the per-node compute the Sthambha planner already measures). This is the move Mesh-LLM *can't* make — it ranks by RTT only. For us, compute-aware ranking extends the planner's intelligence from *layer assignment* to *peer selection*: the mesh prefers to grow toward the nodes that will actually carry layers well.

### 4.4 Trust caveats to design against now
- **Sybil / false advertising** on a public discovery layer remains open even with signed listings (a key can still lie about VRAM). Treat advertised capacity as a *hint*, verify with a liveness + capability probe before assigning real layers.
- **Do not** copy the open-join public mesh. Discovery is public; the mesh is not.

## 5. Layer-package weight distribution (P2 — near drop-in upgrade)

### 5.1 What Mesh-LLM does
A **layer package** is a durable HuggingFace repo: `model-package.json` (manifest with **per-artifact SHA-256 + byte-size**, immutable `hf://ns/repo@revision` pins, relative-only paths) + `shared/` (embeddings/output/metadata GGUFs) + `layers/layer-*.gguf` (per-layer fragments) + optional `projectors/`. A worker fetches **only its assigned stage's fragments.** Real example: `meshllm/Qwen3.6-35B-A3B-UD-Q4_K_XL-layers` — 41 layer fragments (~20.7 GB) + 3 shared GGUFs.

### 5.2 What Nakshatra adopts — adopt it nearly wholesale
Nakshatra already has HTTP byte-range sub-GGUF auto-fetch and `nakshatra.layer_range_start/end` metadata. The layer-package manifest is a **strict superset** of what we have: it adds content-addressing (SHA-256), immutability (`@revision`), and a publish/validate toolchain. This is the closest thing in the whole study to a drop-in:

- Define a `NakshatraPackage` manifest schema (mirror `model-package.json`: schema version, source identity, per-artifact `sha256` + `size`, immutable revision pin).
- A **packager** that fragments a GGUF into `shared/` + `layers/layer-*.gguf` and emits the manifest.
- A **verified fetcher**: given a planner-assigned range `[start,end)`, fetch exactly those fragments, check SHA-256, fail closed on mismatch.

### 5.3 Why it fits the thesis
Fragment fetch is a **one-time staging cost, off the per-token path.** And it makes the planner's measured-compute assignment *directly actionable*: once Sthambha assigns worker B layers 14–27, B pulls exactly those fragments — content-verified, no whole-model download. Discovery (§4) + packages (§5) compose into self-provisioning: a freshly discovered node is told its range and pulls only that range.

### 5.4 Where we go further than Mesh-LLM
Pin fragment integrity to the **same Ed25519 identity** that signs the mesh, so weight provenance and node identity share one trust root. Mesh-LLM's manifest is SHA-256-integrity but not identity-bound; ours can be both.

## 6. Routing entry-proxy (P3 — low effort)

Mesh-LLM runs a per-node proxy on `:9337` that peeks the request `model` field and routes to the host serving it over a tunnel. Nakshatra already exposes `/v1/chat/completions` (Prithvi's L4 talks to the L2 70B through it). Add a thin routing shim so **any node is a valid entry point** and clients address models by name — turning the fixed chain into a real network surface.

**Keep it behind the wall.** Do **not** adopt Mesh-LLM's `api-key: "mesh"` open default. The entry-proxy authenticates against the same Ed25519/Tailscale posture as everything else.

## 7. Wire versioning (P4 — protocol hygiene, do it before strangers join)

Mesh-LLM cleanly separates a **versioned protobuf control plane** (ALPN-negotiated, with a legacy `mesh-llm/0` JSON fallback) from a **raw relay data plane**, and multiplexes control streams by a 1-byte prefix. Nakshatra's gRPC control + UDP fabric already gestures at this two-plane split; v1.0 **formalizes version negotiation** (ALPN-style handshake, explicit protocol version in the gossip frame) so mixed-version peers interoperate. This is cheap now and load-bearing the moment discovery lets a node running last month's build try to join.

## 8. Opt-in verification seam (not the receipts)

The roadmap marker says "opt-in verification." v1.0 ships the **seam**, not the mechanism: the `wanted` field in the listing is a latent demand signal (models a mesh lacks compute for), and the natural supply side is **signed proof-of-work receipts** (`[experiment needed]` in the paper). v1.0 should leave a clean interface where a receipt can be attached to a completed stage, so the incentive layer can land later without re-plumbing. **Build the hook, defer the economics.**

## 9. What we deliberately skip, and why

| Mesh-LLM feature | Decision | Reason |
|---|---|---|
| Mixture-of-Agents (`model:"mesh"`) | **Skip** | Ensemble-quality across *whole models*; orthogonal to our single-stream 70B reasoning role. Spends compute (the scarce resource) and re-opens prompt exposure. Revisit only if Prithvi ever wants cross-model deliberation — and then borrow the in-code deterministic arbiter, not a reducer-LLM. |
| Open public mesh / plaintext-to-relay | **Reject** | Direct conflict with `THREAT_MODEL.md`. Our confidentiality (closed, signed, pinned) is a *feature*, not a gap to close. |
| Stock-llama.cpp + external fragmentation only | **Diverge** | We keep our partial-load patch (`llama_decode_layers`, ~70 LOC) — it's deeper engine work that gives us measured control Mesh-LLM doesn't have. Layer packages sit *on top of* it, not instead of it. |

## 10. Acceptance test (the v1.0 ship gate)

Two machines, **no shared static config**, on separate networks:

1. Both publish a signed `NakshatraListing` (with Ed25519 pubkey) to a Nostr relay.
2. Node A discovers Node B via `--auto`, ranks it (compute-aware), and initiates a join.
3. Admission: token + handshake verifies against B's advertised pubkey; transport comes up on the pinned data plane.
4. The planner assigns layer ranges by measured `Fᵢ`; each node provisions **only its range** from a verified layer package (SHA-256 checked, fail-closed).
5. A request to either node's entry-proxy produces the **byte-identical greedy token** the single-machine `llama-cli` reference produces (the v0.1 parity bar).
6. A mixed-version peer (older protocol) is cleanly rejected *or* negotiated down — never a silent corruption.

If all six hold, hand-wiring is dead and v1.0 ships.

## 11. Prioritized path

1. **§5 Layer packages** first — lowest risk, near drop-in on the existing fetch path, and it's a standalone win even without discovery.
2. **§4 Discovery** — the headline; compose it with §5 so a discovered node self-provisions.
3. **§7 Wire versioning** — land before discovery opens to non-co-located peers.
4. **§6 Entry-proxy** — small shim; can land any time after §4.
5. **§8 Verification seam** — interface only; defer the economics.

Fault tolerance (Petals O(t) dual-cache) is the **immediate successor** to this doc: discovery makes a churning mesh of strangers possible, and a churning mesh *requires* per-token recovery. It is out of scope here but should be the next design doc written.

## 12. Open risks (carried from the study)

- **Sybil/false-advertising on public discovery** survives signed listings — verify capacity, don't trust it.
- **Mesh-LLM has no throughput track record**; its "certified parity" is internal self-validation. We are not inheriting validated performance, only validated *patterns*.
- **`src/` hygiene:** Nakshatra's `src/` still holds the upstream Petals fork while the live engine lives in `experiments/`/`scripts/`/`proto/`. Clean this up before inviting external contributors that discovery will attract.
- **Versioning debt compounds** once strangers join — §7 is not optional polish.
