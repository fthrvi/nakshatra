# v0.0 cb_eval Spike — Plan

**Question.** Can two locally-running llama.cpp processes, each holding the full model, exchange a single hidden-state activation at a fixed layer boundary and produce token output that bit-matches a single-process reference run? If yes, the orchestration / activation-transport / determinism story for v0.1 is de-risked before any C++ patches to `llama_decode` begin.

**Short answer (predicted).** Should work for the *capture* side; *injection* is the open question. `cb_eval` is documented as observation. Whether it permits writing back into the tensor data buffer at the right moment in the graph is what this spike tests. If injection works, v0.1's protocol is validated cheaply. If it doesn't, we know the patched-`llama_decode` work has to come earlier than planned.

---

## 1. What this spike validates

Targets:
- ✅ **Activation transport.** Hidden-state bytes survive the wire correctly (shape, dtype, byte order).
- ✅ **Orchestration.** Worker A (port 50051) → Worker B (port 50052) → driver, fault-free path.
- ✅ **Determinism.** Greedy decoding with a fixed seed produces a token stream that bit-matches a single-process reference. This is `petals-architecture.md` §7's v0.1 success criterion, dragged forward to v0.0 as a forcing function.
- ✅ **`cb_eval` semantics.** Specifically whether `cb_eval` can be used to *modify* a tensor mid-graph (not just observe it) — see §5 below.

Non-targets (deliberately out of scope):
- ❌ Partial loading. Already validated. Both workers load the full GGUF.
- ❌ Cross-vendor execution. v0.3.
- ❌ The patched `llama_decode_layers` API. The whole point of cb_eval is to skip writing this.
- ❌ Authentication, framing robustness, error recovery. This is a spike, not v0.1.
- ❌ Performance. Round-trip time is irrelevant; correctness is the bar.

## 2. Setup

- **Model:** Qwen3-0.6B at `/tmp/nakshatra-test/qwen3-0.6b-full.gguf` (28 transformer blocks, hidden=1024, vocab=151936).
- **Cut point:** between block 13 and block 14 (50/50 split). Each worker runs the full model but only one of them does "real" work for half the layers — see §5.
- **Hosts:** both workers run on localhost. Worker A listens on `127.0.0.1:50051`. Worker B listens on `127.0.0.1:50052`. Driver connects to A, A pushes the cut-point activation to B over a separate TCP connection, B returns the sampled token to the driver.
- **Build:** link against `libllama.so` from `/home/prithvi/llama.cpp/build/`; CPU-only fine. C++ throughout (cb_eval is a C callback; no Python binding exposes it cleanly).

## 3. Pass / fail

The pass criterion is one bit:

> Given prompt P and seed S, the token sampled by the driver from the two-worker pipeline must equal the token sampled by `llama-completion` running the full model in one process with the same P and S, under greedy decoding (`top_k=1`, `temp=0` or `--seed S` with the deterministic sampler chain).

Stretch: tokens 1..5 also match. (If token 1 matches but 2..5 diverge, that suggests a KV-cache state issue rather than a transport issue, which is also informative.)

Failure modes and what each tells us:

| Symptom | Diagnosis |
|---|---|
| Capture bytes don't match captured-but-not-shipped bytes | Transport corruption or fp16↔fp32 confusion |
| Capture works, injection silently has no effect | `cb_eval` is observation-only; the tensor we modify isn't the one the next op reads |
| Tokens diverge after token 1 | KV cache on B is uninitialised or sampler RNG state diverges |
| Both workers crash | We're holding tensor pointers across decode boundaries unsafely |

## 4. Build order

Each step is a checkpoint: stop and commit before moving to the next.

**Step 0 — Reference (≤1 hour).** Run `llama-completion` single-process on the full GGUF with `--seed 42 --top-k 1 --temp 0 --prompt "The capital of France is" --predict 5`. Record the 5 generated tokens. This is the ground truth for §3.

**Step 1 — `cb_eval` sanity check (½ day).** Single-process program: open a context with a `cb_eval` callback that prints every tensor name and shape during one decode of the same prompt. Goal: identify the exact tensor name at the post-block-13 / pre-block-14 cut. Don't modify anything. Confirm the tensor is the right shape (`[hidden=1024, n_tokens=13]` or `[1024, 13]` depending on layout) and named consistently across decode calls.

**Step 2 — Round-trip in one process (½ day).** Same single-process program, but the cb_eval callback now: at the cut tensor, copies bytes out, then on a *second* call sets them back. If the first decoded token matches the reference, we've proven cb_eval-write works. **This is the load-bearing test**: if it fails, we know before any networking that cb_eval is read-only and the spike's approach changes.

**Step 3 — Two processes, no network (½ day).** Same program, but now two contexts in one process, each with its own callback. Worker-A-context decodes layers 0..13, callback dumps the tensor to a file and aborts. Worker-B-context loads the tensor from disk, callback overrides at the same cut tensor, finishes layers 14..27, samples, prints token. If this matches the reference, the cut-point logic is correct.

**Step 4 — Two processes over TCP (1 day).** Split into `worker_a` and `worker_b` binaries (or one binary with `--role a|b`). Wire format:

```
[u32 magic = 0xCBE0]   # spot a wrong-protocol connection cheaply
[u32 length_bytes]     # of the tensor payload
[u16 dtype]            # ggml_type enum
[u16 n_dims]
[u32 dims[n_dims]]     # shape
[bytes data]           # length_bytes of raw tensor data
```

Driver flow:
1. Driver sends prompt to Worker A.
2. Worker A runs decode; cb_eval captures cut tensor, opens TCP to Worker B, sends per the format above.
3. Worker B's cb_eval blocks on TCP receive at the cut tensor; on receive, overrides; finishes decode; samples; sends sampled token ID back to driver.
4. Driver compares token ID to reference.

**Step 5 — Bit-match check (½ day).** Stretch: run for 5 tokens, not 1. Each subsequent token requires Worker A to keep its KV cache for the prompt and run an extra decode for each new token, shipping its activation to Worker B; Worker B keeps its KV cache from layers 14..27 and continues the chain. This is the actual v0.1 inference flow in miniature. If tokens 1..5 all match, we've validated multi-step orchestration.

Total: ~3 days of focused work for someone fluent in C++ and the llama.cpp tree; 1–2 weekends end-to-end as the original memo estimated.

## 5. The central risk: can `cb_eval` modify tensors?

`cb_eval` (`llama.h:350`) is `bool (*cb_eval)(struct ggml_tensor * t, bool ask, void * user_data)`. The signature gives us a `ggml_tensor *` (non-const), so writing to `t->data` is type-permitted. The question is whether it has the intended effect on subsequent graph ops.

Two concerns:
1. **Timing.** When `ask=false`, are we called *after* the tensor is computed but *before* downstream ops read it? Step 2 of §4 directly answers this.
2. **Backend mediation.** On a non-CPU backend, `t->data` may not be a host pointer — it could be a GPU buffer handle. CPU-only build sidesteps this for the spike. v0.1's GPU-backed workers will need to use `ggml_backend_tensor_set` instead. Not a v0.0 concern but worth flagging.

**If Step 2 fails** — i.e., cb_eval-write turns out to be observation-only — the spike pivots: we instead build two cooperating processes that each do their own decode through layer 13, *discarding* the second one's pre-layer-14 inputs and using a custom graph injection point (the `cgraph` callback at a lower level). That's more work but still feasibly a 2-weekend exercise. The cost of finding out early is exactly Step 2's half day.

## 6. What we keep, what we throw away

Throw away when v0.1 starts:
- The cb_eval-based injection (replaced by the patched `llama_decode_layers` entry point).
- Both workers loading the full GGUF (replaced by per-worker sub-GGUFs from the §1.6 validation work).
- Raw TCP wire (replaced by gRPC over Tailscale).
- Hard-coded layer-13 cut (replaced by config-driven layer ranges).

Keep:
- The pass/fail bar (bit-exact greedy decode against single-process reference). This becomes part of `tests/` for v0.1.
- The wire format pattern (shape + dtype + bytes). Maps cleanly to gRPC's bytes field; the schema just gets formalised.
- The driver orchestration pattern. The shape of "client → first worker → ... → last worker → client" is unchanged; only the transport changes.

## 8. Sequencing

This spike is the second pre-implementation gate. Order:

1. ✅ Partial-GGUF load test (`v0.1-validation-partial-gguf.md`, done 2026-05-04).
2. ⏳ This spike.
3. If both pass: commit to v0.1 Path B-prime C++ work with confidence.
4. If this spike fails on cb_eval-write specifically: pivot in this spike, then proceed.
5. If this spike fails on activation bit-match (transport corruption, sampler nondeterminism): fix in spike, then proceed.

The spike does not unblock the v0.1 API gateway work (`v0.1-api-surface.md`) — that can proceed in parallel since it sits above the worker layer.
