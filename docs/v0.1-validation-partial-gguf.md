# Validation Result: Partial GGUF Load Test

**Question.** Does llama.cpp refuse to load a GGUF that contains only some of a model's transformer layers? The original Path B-prime memo (`path-a-vs-path-b-memo.md`) treats this as the load-bearing claim — if true, justifying the C++ patches needed for layer-range execution.

**Short answer.** **The strict claim is falsified.** llama.cpp loads a self-consistent partial GGUF without error and runs forward through it. But the architectural conclusion (Path B-prime) still stands, because the *real* missing abstraction is in the forward-pass shape, not the loader.

---

## Test setup

- Model: Qwen3-0.6B (HuggingFace `Qwen/Qwen3-0.6B`), 28 transformer blocks, 311 tensors total.
- Build: llama.cpp at commit `c46583b` (v0.9.8), CPU-only build (`build/bin/llama-completion`).
- Procedure:
  1. Convert HF → GGUF with `convert_hf_to_gguf.py` → `qwen3-0.6b-full.gguf` (1.5 GB, 311 tensors).
  2. Construct a sub-GGUF with `gguf-py/`: keep `blk.0..blk.13`, `token_embd`, `output_norm`, `output`; drop `blk.14..blk.27`. **Override `qwen3.block_count` from 28 to 14** to keep the file self-consistent. → `qwen3-0.6b-half.gguf` (1.07 GB, 157 tensors). Construction script: `/tmp/nakshatra-test/make_sub_gguf.py` (~110 lines).
  3. Run `llama-completion -p "The capital of France is" -n 5` against both GGUFs.

## Observed result

| | Full (28 blocks) | Half (14 blocks) |
|---|---|---|
| Load | clean | clean (no warnings, no errors) |
| Exit code | 0 | 0 |
| Memory (host) | 6214 MiB | 3554 MiB |
| Decode speed | 20.18 t/s | 32.20 t/s |
| Generated tokens | `<think>\nOkay, the` | `eightheenth_re` (nonsense) |

The half GGUF parsed cleanly with `qwen3.block_count u32 = 14` and 157 tensors. llama.cpp treated it as a coherent 14-layer Qwen3 model.

## Why the output is garbage (and why that's correct)

Layers 0–13 of a 28-layer Qwen3 produce hidden states meant to be transformed by *14 more layers* before `output_norm` and the LM head read them. We bypassed those 14 layers and let `output_norm` + LM head read layer-13 activations directly. The result is nonsense tokens — exactly what we'd expect when you feed a layer's outputs into a head that wasn't trained to read them.

This is not a bug. It is the *signature* of what's missing. We need an inference entry point that:
- **Skips the embed lookup** when receiving hidden states from another worker (rather than token IDs).
- **Skips `output_norm` and the LM head** when sending hidden states to another worker (rather than producing logits).

Vanilla llama.cpp doesn't expose either skip. That is the real missing abstraction.

## Implication for Path B-prime

The original `path-a-vs-path-b-memo.md` justified C++ patches as "the loader can't handle partial models." That framing was wrong; the loader handles them fine. The accurate framing:

> **Path B-prime is needed because llama.cpp's forward-pass entry points are fixed at the input boundary (always embed) and the output boundary (always project to logits). For pipeline-parallel execution across workers, intermediate workers need a `forward(hidden_in, layer_start, layer_end) → hidden_out` shape that bypasses both boundaries.**

This sharpens *where* the C++ work is, but does not shrink the original effort estimate:
- **Loader patches:** none needed. Sub-GGUF production is a Python concern (~110 lines, demonstrated).
- **Forward-pass patches:** add `llama_decode_layers(...)` that takes a hidden-state tensor instead of tokens, runs only the requested layer range, returns hidden states. The patches are concentrated in `llama_decode_internal` and the graph-build path.
- **Effort estimate:** unchanged from the original memo (10–14 weeks experienced / 15–28 newcomer). Re-reading `path-a-vs-path-b-memo.md` §5.4, the breakdown was already pre-split-script + patched-decode + DLPack shim + gRPC integration — no loader patches were ever included. So removing "loader work" from the rationale doesn't move the dial; what it does is concentrate confidence around the patched-decode line item, which is the dominant 6–10 week chunk.

## Implication for Path A

Path A (wrap `rpc-server`) is *still* infeasible for the original reason: `rpc-server` operates at GGML primitive level (`RPC_CMD_GRAPH_COMPUTE`, etc.), not at transformer-layer level. Even with our sub-GGUF discovery, an `rpc-server`-based worker would need to know about layer ranges and skip-boundaries, which means re-implementing the layer-range concept inside or alongside the RPC layer. No change to the Path A verdict.

## Open follow-up (deferred, not blocking)

The next validation step from the v0.1 plan is the **`cb_eval` spike** — a 1–2 weekend prototype where two workers each load the *full* GGUF, exchange activations through llama.cpp's per-tensor `cb_eval` callback, and produce a token stream that matches a single-machine reference run. That validates the *protocol/orchestration* design without needing the C++ forward-pass patches yet. This test (partial-GGUF load) doesn't change anything about the cb_eval spike's design.
