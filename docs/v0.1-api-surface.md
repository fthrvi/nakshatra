# Nakshatra v0.1 User-Facing API Surface

**Question.** What HTTP API does a Nakshatra cluster expose to its callers in v0.1, and how does that surface translate to the internal gRPC fan-out across workers?

**Short answer.** v0.1 exposes an **Ollama-compatible HTTP gateway** (`/api/chat`, `/api/generate`, `/api/tags`) sitting in front of the internal gRPC-over-Tailscale worker mesh. The gateway is a thin translator: it parses Ollama's wire format, assembles a per-request inference plan, fans tokens out through the gRPC pipeline, and streams JSON-line chunks back. Internal gRPC stays an implementation detail; end users never touch it. This shape is chosen because Nakshatra's first real consumer (Prithvi) already speaks Ollama, and the broader Ollama ecosystem (Open WebUI, LangChain, Cursor, the Ollama CLI itself) becomes drop-in compatible for free.

---

## 1. Why HTTP at the edge, not gRPC

Internal worker-to-worker traffic is gRPC over Tailscale (per `path-a-vs-path-b-memo.md` and the v0.1 architecture decision) — it carries DLPack-serialised activation tensors, needs HTTP/2 multiplexing for the bidi activation stream, and benefits from Protobuf's tight binary framing. None of that buys end users anything. Their needs are different:

- They want a familiar request shape they can reach with `curl`, `httpx`, or a stock SDK.
- They want server-streamed tokens visible in a debugger, not a Protobuf packed struct.
- They don't care about the activation wire — they care about messages and tokens.

So HTTP at the edge, gRPC inside. The two are not in tension; they sit on opposite sides of the gateway.

## 2. Why Ollama-compat specifically (not OpenAI-compat)

Both shapes have ecosystem reach. The argument for Ollama:

1. **First consumer already speaks it.** Prithvi's `neuron-net/gateway/router.py:857` calls `http://host:port/api/chat` directly. Swapping in Nakshatra is a config change, not a code change.
2. **It's a superset, not a subset.** Ollama exposes `num_predict`, `repeat_penalty`, `top_k`, `tfs_z` as request-time options. OpenAI's chat completions API doesn't surface those. Prithvi uses them dynamically (`router.py:482-514` shapes them from its consciousness state). Going OpenAI-only would force every caller into the lowest common denominator.
3. **Tool-use stays simple.** Prithvi parses custom `[TOOL: ...]` markers post-stream. OpenAI function-calling is a richer protocol Nakshatra would have to implement *and* Prithvi would have to ignore. Skip it.
4. **Local-model framing matches the product.** Ollama's API was designed for "this is the model running on your machines." OpenAI's was designed for "this is OpenAI's hosted service." Nakshatra is the first.

Future versions can add a small OpenAI-compat shim for users who want it. v0.1 doesn't.

## 3. Endpoints v0.1 implements

| Endpoint | Method | Purpose | Streaming |
|---|---|---|---|
| `/api/chat` | POST | Multi-turn chat completion | yes (line-delimited JSON) |
| `/api/generate` | POST | Single-prompt completion | yes |
| `/api/tags` | GET | List models the cluster can serve | n/a |
| `/api/version` | GET | Gateway version + protocol info | n/a |

Out of scope for v0.1: `/api/embeddings`, `/api/show`, `/api/pull`, `/api/push`, `/api/create`, `/api/copy`, `/api/delete`. Embeddings could land in v0.2; the model-management endpoints are a Prithvi-style coordination concern that does not belong in Nakshatra's contract.

## 4. Request and response shapes

**`POST /api/chat`** — request:

```json
{
  "model": "qwen3:14b",
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "..."}
  ],
  "stream": true,
  "options": {
    "num_predict": 512,
    "temperature": 0.7,
    "top_p": 0.9,
    "top_k": 40,
    "repeat_penalty": 1.1
  }
}
```

Streaming response — line-delimited JSON, one frame per token:

```json
{"model":"qwen3:14b","created_at":"...","message":{"role":"assistant","content":"Hello"},"done":false}
{"model":"qwen3:14b","created_at":"...","message":{"role":"assistant","content":" there"},"done":false}
...
{"model":"qwen3:14b","created_at":"...","message":{"role":"assistant","content":""},"done":true,"total_duration":...,"eval_count":...}
```

Non-streaming returns a single object with `done: true` and the full assembled message.

The terminal `done: true` frame must include `total_duration`, `prompt_eval_count`, `prompt_eval_duration`, `eval_count`, `eval_duration` — Prithvi's `router.py` reads these to update its node-affinity bookkeeping.

## 5. Gateway → internal gRPC translation

Per request:

1. **Parse and validate.** Reject unknown options rather than silently dropping them — saves users the "why isn't my temperature working" debugging.
2. **Resolve model → pipeline.** Look up the static YAML config for which workers hold which layer ranges of the requested model. v0.1 has no dynamic placement.
3. **Tokenise on the gateway.** Hidden-state pipelines need pre-tokenised input; the first worker should receive token IDs, not raw text. Tokeniser ships with the gateway (loaded from the same GGUF metadata).
4. **Open a single bidi gRPC stream** to the first worker in the pipeline. Activation tensors flow worker → worker through the chain (per `petals-architecture.md` §4).
5. **Sample at the last worker.** Final-layer logits stay on whichever worker holds the LM head; sampling happens there so we avoid shipping `[batch, vocab]` tensors back to the gateway every step. The last worker streams sampled token IDs back to the gateway.
6. **Detokenise and frame.** Gateway converts token IDs → string deltas → JSON-line chunks → HTTP body.

Open question: **where exactly does the sampling state (RNG, repetition tracking) live?** Two options — last-worker-side (simple, but requires the worker to know about sampling parameters per request) or gateway-side (cleaner separation, but means shipping logits over the wire). Default to last-worker-side for v0.1; revisit if logit-shipping turns out to be cheap enough (vocab × fp16 ≈ 256 KB for Llama-3, not nothing but not catastrophic).

## 6. Auth, sessions, errors

- **Auth.** v0.1 trust model is "trusted workers on a private network." Gateway accepts any Bearer token and logs it for accounting; it does not validate. Prithvi's wallet validation happens upstream of Nakshatra. A real auth story comes with v0.5+ when the trust boundary widens.
- **Sessions.** Stateless. Every `/api/chat` request carries the full message history. KV-cache affinity (sticky-routing repeated prefixes to the same pipeline) is a future optimisation; v0.1 rebuilds the cache from scratch per request, which matches the deferred-features list.
- **Errors.** Worker failures during a request → fail+full-retry (3x, per v0.1 architecture). The HTTP response is a final JSON line with `done: true, error: "..."`. We do not partial-stream then 500 mid-body — that breaks every client.

## 7. Out of scope for v0.1

- **Image multimodal.** Prithvi's `images` field gets a 400 response with an explicit `"images not supported in v0.1"` error. No silent drop.
- **OpenAI function-calling / tool_calls.** Prithvi parses its own `[TOOL: ...]` markers post-stream. No v0.1 caller needs server-side tool framing.
- **Embeddings.** No v0.1 caller needs them.
- **Concurrent requests sharing a pipeline.** v0.1 serialises requests per pipeline. Concurrency is across pipelines, not within one. v0.5+ adds intra-pipeline batching.

## 8. Open questions to resolve before implementation

1. **Sampling location.** Last-worker vs gateway. Defaulting to last-worker-side; needs a 30-line spike before locking in.
2. **Tokeniser hosting.** Gateway loads the tokeniser from the GGUF — confirms the gateway needs read access to the same GGUF files the workers do. Does it load a full copy, or just the tokeniser blob?
3. **`/api/tags` source.** Is the list of available models the union of what's in the static YAML, or does the gateway probe each worker's `/health`? Probably YAML for v0.1 — workers can be down without removing the model from the catalogue.
4. **Concurrency model.** asyncio + httpx for inbound HTTP, grpc.aio for outbound gRPC. Both async, same loop. No threading.
