# Nakshatra v0.0 — Pre-Implementation Validation Plan

**Date:** 2026-05-06
**Status:** Test plan — not yet executed. Sibling to [`petals-architecture.md`](petals-architecture.md), [`path-a-vs-path-b-memo.md`](path-a-vs-path-b-memo.md), and [`north-star.md`](north-star.md). Operationalises the empirical experiments referenced in `path-a-vs-path-b-memo.md` §1.6 and §5.7, and `petals-architecture.md` §8.7.

---

## Why this doc exists

`path-a-vs-path-b-memo.md` recommends **Path B-prime** for the v0.1 LlamaCppBackend: bypass llama.cpp's RPC layer, reuse its per-architecture graph builder, patch `llama_decode` to accept hidden-state input/output. The estimated effort is **10–14 weeks of focused C++ work** (memo §5.4), with significant ramp-up cost for an engineer new to llama.cpp internals (15–28 weeks for first-time, memo §5.4).

That is a real budget. Before committing to it, two cheap experiments must pass. Both are explicitly called out in the existing docs (memo §1.6 + §5.7, architecture §8.7) but not yet executed. This plan turns them into something runnable on hardware Biswa has access to today.

If both experiments pass, we commit to v0.1 with confidence. If either fails, we have spent ≤1 week of effort instead of ≤14 weeks learning that the architecture needs to change.

---

## The decision being validated

The Path B-prime memo rests on two empirically-falsifiable claims:

1. **llama.cpp has no usable partial-model load path.** The memo (§1.5) infers this from reading `llama.h` and `src/llama.cpp`. If wrong, much of the v0.1 design changes.
2. **Per-layer hidden-state injection and extraction works through `cb_eval`** well enough to demonstrate end-to-end distributed inference of an unmodified model. If wrong, even the v0.0 spike is unfeasible and the patched-`llama_decode` work cannot be deferred.

Phase 0a tests claim 1. Phase 0b tests claim 2. Both must pass to greenlight v0.1.

## Phase 0a — Partial-GGUF load test

**STATUS (2026-05-06): RESOLVED.** Outcome was the "Hard error" branch — loader rejects partial GGUFs with `missing tensor 'blk.20.attn_norm.weight'` (or `'output_norm.weight'` if the output head is also dropped). Path B-prime confirmed. Experiment ran on the lab Mac `bishwa`, not the home PC (the 70B GGUF lives there, not on the home PC as the original plan assumed). Findings: [`experiments/v0.0/partial_gguf_findings.md`](../experiments/v0.0/partial_gguf_findings.md). The original plan text follows for reference.

**Source:** `path-a-vs-path-b-memo.md` §1.6.
**Time budget:** half a day. (Actual: ~2 hours including SSH key setup and venv install on the lab Mac.)
**Hardware:** home PC alone (Linux, RX 9070 XT, ROCm). One machine, no networking required. (Actual: ran on lab Mac `bishwa` because the 70B GGUF was there and the home PC's GPU was actively serving ollama + Prithvi-daemon.)

### Goal

Falsify (or confirm) the claim that llama.cpp's loader rejects a GGUF that contains only a subset of a model's layers. This claim is the load-bearing assumption behind Path B-prime: if llama.cpp can already load a partial model, the v0.1 patched-`llama_decode` work shrinks dramatically.

### Inputs

- `Llama-3.3-70B-Instruct-Q4_K_M.gguf` — `path-a-vs-path-b-memo.md` §1.6 asserts this is on the home PC. **Verify with `ls` before starting.** If absent, substitute any locally-present GGUF (Llama-3-8B works equally well for this experiment; the question is about the loader, not the model size).
- `~/llama.cpp` source tree — clone if not present; build `llama-cli` with the ROCm backend.

### Steps

1. **Verify inputs.**
   - `ls -lh ~/path/to/Llama-3.3-70B-Instruct-Q4_K_M.gguf`
   - `cd ~/llama.cpp && cmake -B build -DGGML_HIPBLAS=ON && cmake --build build -j` (or whatever the current ROCm build incantation is — check the upstream README before committing to flags)
   - `./build/bin/llama-cli --version`

2. **Write the partial-GGUF script.** New file: `experiments/v0.0/partial_gguf.py`. Uses `~/llama.cpp/gguf-py/` to:
   - Parse the full GGUF.
   - Identify tensors belonging to layers `[0, 20)` plus the embedding (`token_embd.weight`) plus any RoPE / rope-freqs metadata tensors plus the output norm + lm_head (or *omit* lm_head — see "outcome interpretation" below).
   - Write a sub-GGUF (`partial.gguf`) containing only those tensors and the original metadata header.
   - Print which tensors were included and which were dropped so the resulting file is auditable.

3. **Attempt to load the sub-GGUF.**
   - `./build/bin/llama-cli -m partial.gguf -p "Hello" -n 1`
   - Capture full stdout + stderr to `experiments/v0.0/partial_gguf_run.log`.

4. **Record the outcome** in `experiments/v0.0/partial_gguf_findings.md`: which of the three branches below fired, the exact error message, and a short interpretation.

### Falsifiable outcomes

| Outcome | What it means | Next step |
|---|---|---|
| Hard error at load (e.g. `missing tensor: blk.20.attn_q.weight`) | Path B-prime confirmed. Patched `llama_decode` is the right path. | Proceed to Phase 0b. |
| Loads metadata but errors at inference time | Most informative — tells us *exactly where* in the pipeline complete-model presence is assumed. Narrows the patch scope and likely reduces v0.1 estimate. | Note the error site, proceed to Phase 0b, revise `path-a-vs-path-b-memo.md` §5.4 estimate downward. |
| Loads cleanly and produces a token | llama.cpp has partial-load capability the memo missed. Architecture decision must be re-examined. | **Stop the validation here.** Re-read `src/llama.cpp` and `src/llama-model-loader.cpp`. Rewrite memo §1.5 with corrected understanding. Possibly redesign v0.1 around the discovered capability. |

### Failure mode

If the script blows past one day on GGUF parsing, something is wrong with the script or the GGUF format assumptions. Stop, write up what was tried, and ask. Do not push through.

## Phase 0b — v0.0 cb_eval spike

**Source:** `path-a-vs-path-b-memo.md` §5.7.
**Time budget:** 1–2 weekends.
**Hardware:** two machines, ideally same OS + same vendor + same backend so the variable being tested is the *protocol*, not the cross-vendor compatibility. Recommended: two of the lab iMacs (Intel + Radeon Pro 5700 XT, Vulkan-via-MoltenVK), per `petals-architecture.md` §7.1's first MVP step.

### Goal

Prove the orchestration protocol and activation transport on real hardware *before* paying any v0.1 C++ cost. Each worker holds the **full** model — partial loading is deferred to v0.1. What this validates is the wire format, the handshake, the callback-based hidden-state injection and extraction, and the round-trip token correctness.

If this succeeds, v0.1 inherits a known-working orchestration design and only needs to swap the "full-model + cb_eval hack" for "partial-model + patched `llama_decode`."

### Architecture

- **Worker 1** loads `Llama-2-7B-Q4_K_M` fully. Runs `llama_decode` with a `cb_eval` callback that, when the post-layer-15 hidden state appears (layer count is 32 for Llama-2-7B; mid-point is 16), captures it, ships it over TCP to Worker 2, then aborts the rest of the decode.
- **Worker 2** loads the same model fully. Receives the hidden state from Worker 1. Runs `llama_decode` with a `cb_eval` callback that, at layer 16's input, overwrites the hidden state with the bytes received from Worker 1. Continues to logits, samples one token, sends it back to the orchestrator.
- **Orchestrator** is a thin Python script that holds the prompt, drives Worker 1 + Worker 2, collects the token, and runs the comparison test below.

### Implementation language

Try `llama-cpp-python` first. Its `Llama(...)` constructor exposes `cb_eval` (and similar low-level GGML callbacks) via ctypes. If the Python binding does not let you both *read* and *return-modified* hidden state from the callback (i.e. it is observe-only), fall back to a thin C++ binary that links against llama.cpp.

The Python path collapses this from "weekends" to "an evening." A 30-minute spike against `llama-cpp-python` to confirm `cb_eval` ergonomics is the first task — do that *before* writing anything else.

### Falsifiable success criterion

This is the same shape as `petals-architecture.md` §7.1's v0.1 success criterion, restated for v0.0:

- **Setup:** fixed prompt `"The capital of France is"`, greedy decoding (`temperature=0`, `top_k=1`, `top_p=1.0`), fixed seed if the sampling implementation accepts one (greedy with `top_k=1` should be deterministic regardless), Q4_K_M quantization, identical GGUF on both workers.
- **Reference run:** single-machine `llama-cli` with the same model, same sampling, produces top-1 next-token T_ref (capture both the token id and the decoded string for the log).
- **Distributed run:** the two-iMac spike produces top-1 next-token T_dist.
- **Pass:** `T_ref.token_id == T_dist.token_id`. Compare token ids, not decoded strings — different ids can decode to the same string in some tokenizers.
- **Fail:** they differ. Bitwise-equal hidden states are NOT required (numerical drift through different kernel paths is allowed within ε), but the top-1 prediction must match. If it does not, the spike has a correctness bug — most likely in activation serialization (dtype mismatch, wrong shape, byte order, NaN propagation through the callback).

### Deliverables

- `experiments/v0.0/spike_worker.py` (or `.cpp` if Python falls short)
- `experiments/v0.0/spike_orchestrate.py` — drives both workers, runs the test
- `experiments/v0.0/spike_findings.md` — pass/fail, the token ids observed, any debugging notes

### Failure mode

If 0b blows past three weekends, the spike is fighting `cb_eval`'s ergonomic limits. That is itself a useful signal — it means the v0.1 patched-`llama_decode` work is unavoidable, which is what `path-a-vs-path-b-memo.md` §5.5 already predicts. Document the friction and stop; the v0.1 commitment is sound either way.

## Phase 0 decision gate

Once both phases have run, the gate logic is:

| 0a outcome | 0b outcome | Decision |
|---|---|---|
| Hard error confirmed | ✅ pass | **Commit to v0.1 Path B-prime.** Write the v0.1 implementation plan as a separate doc. Start C++ work on home PC + 4th-PC pair (when 4th PC is built). |
| Hard error confirmed | ❌ fail | Diagnose 0b first. Wire-format bug → fix and rerun. `cb_eval` cannot return modified state → write up and proceed to v0.1 anyway, since v0.1's patched `llama_decode` does not depend on `cb_eval`. |
| Inference-time error | run 0b anyway | Phase 0a's "loads but errors at inference" outcome is information you want before sizing v0.1; 0b is independent. Update memo's effort estimate based on where the inference-time error occurred. |
| Loads cleanly | skip 0b | **Architecture re-examination.** Rewrite memo §1.5. Possibly redesign v0.1. Do not run 0b until the new architecture is settled — the spike's design assumes partial loading is impossible. |

## v0.1 acceptance test — forward-looking sketch

Once v0.1 ships (10–14 weeks of C++ work after the gate), the ship criterion (`petals-architecture.md` §7.1) becomes:

- Same fixed-prompt + greedy-decoding setup as Phase 0b.
- Reference: single-machine `llama-cli` with the chosen v0.1 model GGUF.
- Distributed: Nakshatra running two-iMac mode with **per-worker sub-GGUFs** (this is the difference from 0b — partial loading is now real).
- Pass: top-1 first-generated token matches reference.

The test harness from Phase 0b can be reused unchanged — the only thing that differs at v0.1 is the worker implementation, not the orchestrator. Reuse it.

The v0.2 and v0.3 acceptance tests (per `petals-architecture.md` §7.2 and §7.3) follow the same shape with one variable changed at a time: v0.2 adds Linux + ROCm to the chain; v0.3 adds CUDA on a rented cloud GPU.

## Open prerequisites before starting

These do not block writing this doc, but they block running the experiments. Resolve before Phase 0a.

- **Llama-3.3-70B-Instruct-Q4_K_M.gguf actually present on the home PC?** Memo §1.6 asserts it. Verify with `ls` before writing the script. If absent, swap to any locally-present GGUF — the experiment is about the loader, not the specific model.
- **llama.cpp ROCm build works on RX 9070 XT?** This is the toolchain used for Phase 0a and (eventually) the home-PC half of v0.2. Worth a 10-minute build-and-run test before sinking time into the partial-GGUF script.
- **Lab iMac access for the duration of the spike.** You are at the lab on 2026-05-06. Can two iMacs be commandeered for a weekend, or does this need scheduling with whoever owns them?
- **Tailscale already on the lab machines?** If not, that is a 10-minute prereq for the network leg of Phase 0b.
- **`llama-cpp-python` `cb_eval` ergonomics.** A 30-minute spike confirming the binding lets you both observe and *return-modify* hidden state in the callback. If it does not, plan for the C++ fallback before starting the spike proper.
