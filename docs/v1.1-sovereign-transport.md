# Nakshatra v1.1 — Sovereign cross-network transport

**Date:** 2026-06-11. **Status:** design proposal (the next build). Successor to
[`v1.0-discovery-and-distribution.md`](v1.0-discovery-and-distribution.md) and the
empirical [`cross-machine-validation.md`](cross-machine-validation.md).

> **▶ RESUME HERE next session.** v1.0 (P1–P4 + Nostr transport) is built, merged,
> and proven end-to-end across two heterogeneous machines on a *shared tailnet*
> (discover → pin → self-provision → negotiate → forward → byte-identical first
> token). This doc designs the one piece that turns "our tailnet mesh" into "a
> network strangers on *different* networks can join."

---

## 1. The one-sentence problem

Discovery already crosses any network boundary — Nostr is public, so a node on
**our** tailnet and a node on a **stranger's** network find each other, rank, and
pin with zero shared infrastructure. But the **data plane** (weight fetch +
activation forwarding) currently reaches another machine *only because Tailscale
provides the overlay* — addressing + NAT traversal for free. Two peers on
**different** tailnets / behind **different** NATs have **no way to connect**.
That is the last ceiling: **discovery is public; the transport is not.**

(Empirically confirmed in the cross-machine run: the data plane reached Mac4
purely because both were on one tailnet. Remove that and there is no path.)

## 2. The goal

A **per-peer encrypted tunnel, raised on demand, keyed by the pinned Ed25519
identity** from the Nostr listing — so any two admitted identities can move
activations between themselves across arbitrary networks, **without a shared
VPN**. Tailscale becomes one *optional* convenience, not the architecture.

Design rule (unchanged): **borrow the mechanism, keep the wall.** Discovery is
public gossip; the tunnel is closed, mutually authenticated against the pinned
key, and carries the same gRPC data plane we already run.

## 3. The layered flow (what changes, what doesn't)

```
1. Discovery (Nostr, public)            — UNCHANGED. listing carries Ed25519 pubkey,
                                          endpoint hint, + NEW: connection info.
2. Admission (pin Ed25519 key)          — UNCHANGED.
3. Transport bringup (NEW)              — raise a tunnel to the pinned peer:
     a. exchange connection params (via the listing / a Nostr DM / the pillar)
     b. NAT-traverse: hole-punch, else relay-fallback
     c. authenticate the tunnel against the pinned Ed25519 identity
4. Data plane (gRPC over the tunnel)    — UNCHANGED bytes; now rides the tunnel
                                          instead of assuming a shared tailnet.
5. Drift-class gate (from the gauge)    — chain only forms within one engine-build
                                          drift class (see cross-machine-validation §2a).
```

Only step 3 is new. Steps 1–2, 4 already exist; step 5 is the gauge (built).

## 4. Transport options (decision needed next session)

| Option | What | Pro | Con |
|---|---|---|---|
| **A. WireGuard, pillar-issued** | Per-peer WG tunnel; the **Pillar already issues WG peers keyed to identity** (Mode-B `POST /peer/personal`) | Reuses a *built* building block; kernel-fast; simple, audited crypto | WG needs a reachable endpoint or a relay for hard NATs; key/endpoint exchange to design |
| **B. QUIC + hole-punching (iroh-style)** | Exactly Mesh-LLM's path: QUIC with NAT hole-punching + relay fallback | Best NAT traversal; relay fallback built into the model; library exists (iroh) | New dependency + Rust/FFI surface; more moving parts than WG |
| **C. Hybrid** | WG when a direct endpoint exists (LAN/port-forward/tailnet), QUIC/relay when not | Best of both; graceful | Two transports to maintain |

**Leaning A→C:** start with **WireGuard pillar-issued** (we own the building block —
see `project_personal_device_join_path` / Mode B), since the §10 mechanics are
already proven over a WG-like overlay (Tailscale *is* WireGuard). Add a **relay
fallback** (a node we run, like Mesh-LLM's Fly.io relay, reachable by both peers)
for hard NATs. Evaluate iroh-QUIC only if WG hole-punching proves insufficient.
Decide with a connectivity spike next session.

## 5. What the listing must carry (P1 extension)

`NakshatraListing` today carries `ed25519_pubkey_hex`, `endpoint_hint`,
`serving`, `measured_decode_ms_per_layer`, `supported_protocol`. Add (all
optional, signed under the same Ed25519 self-signature):

- `wg_pubkey` — the peer's WireGuard public key (option A).
- `transport` — advertised transports, e.g. `["wg", "quic"]` (negotiate like
  `supported_protocol`).
- `relay_hint` — a relay the peer can be reached through if direct fails.
- `drift_class` — the gauge fingerprint hash (so discovery pre-filters to a
  bit-deterministic class; from `cross-machine-validation.md` §4). *This is the
  cheapest immediate win and can land independently of the transport.*

The listing stays the bootstrap for *everything*: identity (pin), where (endpoint/
relay), how (transport), and what-class (drift). The tunnel carries the rest.

## 6. Authentication — bind the tunnel to the pinned identity  ✅ BUILT

The tunnel MUST be mutually authenticated against the **Ed25519 mesh key the
listing was pinned to** — not a separate trust root. The first message over the
tunnel is an Ed25519-signed **challenge-response binding the transport session to
the pinned key**. A tunnel that comes up but fails the binding is dropped —
discovery is public, admission is not. (Same posture as the existing SPKI/TLS
pinning, generalized off the tailnet.)

**Built (2026-06-11): `scripts/transport/identity_handshake.py`.** Transport-
agnostic mutual handshake — each side signs `nakshatra-tunnel-v1 | role | nonces |
session_binding` and verifies the peer's proof against the *pinned* key. The
`session_binding` ties the proof to the underlying channel (a WG key-pair hash, a
TLS exporter, …) so a relay that forwards bytes can't substitute itself. Reuses
the repo's one Ed25519 scheme. 8 tests incl. real-socket mutual auth, MITM
(wrong-pin) rejection, and anti-relay binding-mismatch. This is the
Nakshatra-specific layer; it sits on top of whatever transport §4 chooses.

## 7. Acceptance test (the v1.1 ship gate)

Two machines on **genuinely different networks** (not one tailnet — e.g. two
different home/LTE networks, or one tailnet-off):

1. Both publish signed listings (with `wg_pubkey`/`transport`/`drift_class`) to a
   public-or-self-hosted Nostr relay.
2. Node A discovers B, ranks, pins B's Ed25519 key, reads its transport + drift
   class.
3. A raises a tunnel to B (hole-punch or relay), authenticated against the pinned
   key. **No shared VPN.**
4. The drift-class gate confirms A and B share an engine build (else refuse a
   deterministic chain).
5. The gRPC chain runs over the tunnel → byte-identical first token (`12366
   ' Paris'`).

If that holds with Tailscale **off** on at least one side, the ceiling is gone.

## 8. Phasing (smallest valuable first)

1. **`drift_class` in the listing + routing filter** — ✅ DONE (2026-06-11): `NakshatraListing.drift_class` (signed), `resolve_serving_peer`/`route_or_local` `require_drift_class` filter (same-class-only for deterministic chains), `discover --drift-class`. 3 routing tests.
2. **Identity-bound tunnel handshake (§6)** — ✅ DONE (2026-06-11):
   `scripts/transport/identity_handshake.py`. The auth layer is ready; it binds
   any transport to the pinned key.
3. **Connectivity spike** — ⚠️ partly run + **a finding that reorders 4 & 5**
   (see §8.2 below). Prove the data plane runs without the shared tailnet.
4. **Relay fallback FIRST** (moved up) — the recon (§8.2) shows the common case is
   *both peers behind NAT on different networks*, where direct WG is impossible.
   A self-hosted relay (Mesh-LLM-style) is therefore **mandatory, not a fallback**.
5. **Pillar-issued WG bringup** — wire the listing's `wg_pubkey`/`relay_hint` to
   the Mode-B peering so the tunnel raises automatically on admission, using the
   relay when direct fails.
6. **QUIC/iroh** — evaluate iff WG-over-relay proves insufficient.

### 8.2 — connectivity-spike recon finding (2026-06-11)

Attempted the Tailscale-off WG spike (this Linux box ↔ Mac4). **A true off-tailnet
WG tunnel is not achievable in the current environment, for reasons that are
themselves the finding:**

- **No passwordless sudo** on either machine → can't create a kernel WG interface.
- **No WireGuard tooling** installed (and installing needs sudo); no userspace
  wireguard-go/boringtun either (and a TUN still needs root).
- **The two machines cannot reach each other off the tailnet at all:** this box is
  NAT'd on `10.0.0.0/24`, Mac4 is on a university LAN `129.24.97.0/24`, and
  `129.24.97.77` is **not directly reachable** from here. The tailnet's NAT
  traversal is the *only* path between them.

That third point is the architectural result: **the common case is both peers
behind NAT on different networks, where plain WireGuard-with-a-static-endpoint
cannot connect.** So the design's earlier "lean WireGuard-direct first" is wrong
for the real world — **a relay (or hole-punching) is mandatory**, which is exactly
why Mesh-LLM ships a relay + iroh hole-punching. Phasing reordered above: stand up
the relay first; treat direct WG as the *optimisation* for when a peer does have a
reachable endpoint.

What the spike DID land: the **identity-bound handshake (§6)** — the auth that
gates any such tunnel — is built + tested, and is transport-agnostic, so it's
ready to drop onto the relay/WG path whenever the connectivity substrate exists.

### 8.4 — rendezvous relay BUILT + proven on the Pillar (2026-06-11)

`scripts/transport/relay.py` — an **untrusted rendezvous relay**: two NAT'd peers
each connect *out* with a shared `rendezvous_id`; the relay pairs them and pipes
raw bytes (never interprets them). The peers run the identity handshake END-TO-END
over the relayed pipe, so the relay can forward but **cannot impersonate** (it
can't forge the pinned-key proof). 3 tests + the 8 handshake tests.

**Proven live:** deployed `relay.py` on the **Pillar** (umbrel Pi, always-on) and
ran it across two machines on **different networks** — this box (home LAN) and
**Mac4** (university LAN). Each connected *out* to the Pillar; the relay paired
them; the mutual identity handshake succeeded end-to-end (`HANDSHAKE_OK`, each
pinned to the other's Ed25519 key). **This box's leg ran OFF-TAILNET over the
Pillar's public IPv6** (`2601:…::a022`); Mac4's leg used the tailnet (its only
path — it's IPv4-only behind university NAT).

This is the Bitcoin "public listening node" + WebRTC-TURN model, on real infra.

### 8.5 — the DATA PLANE traversed the relay → token (2026-06-11) ✅

`mux_tunnel.py` (many streams over one authenticated pipe) + `tunnel_endpoint.py`
(relay.connect → handshake → mux). Ran the **real gRPC chain through the Pillar
relay**: worker A on this box `[0,8)`, worker B on **Mac4** `[8,16)` addressed
*only* at the local tunnel listener `127.0.0.1:15531` → mux → Pillar relay →
Mac4:5531. So the client's calls to B AND worker A's activation-forward to B all
crossed the **untrusted relay**, multiplexed over one identity-pinned pipe. Result:
`step 1: id=12366 ' Paris'` — the §10 parity token, second half computed on Mac4
reached entirely through the relay. **The sovereign transport carries real
inference.**

**Still to do for a fully-sovereign data plane:** (b) an **encryption layer**
(WireGuard/Noise) over the pipe so the relay sees only ciphertext (it currently
forwards plaintext-after-auth — the handshake authenticates but doesn't encrypt
the stream); (c) a **public-IPv4 relay host** (a VPS) so *both* legs are
off-tailnet — the Pillar can't serve Mac4 off-tailnet (Mac4 is IPv4-only, the
Pillar's public address is IPv6, no shared path). With (b)+(c) the §7 acceptance
test (two different-network machines, Tailscale OFF, byte-identical token) is met.

## 9. Relationship to the rest of the roadmap

- **Fault tolerance** (the other named successor) composes here: a churning mesh
  of strangers on different networks needs Petals-style O(t) recovery, and
  recovery must stay **same-drift-class** (the gauge) AND re-establish the tunnel.
- **Mesh-LLM** solves this with **iroh-QUIC** — see `vs-mesh-llm.md`. This is the
  one area they are ahead; v1.1 closes it on our terms (pinned-identity-keyed, not
  open-join).
- **Prithvi/Sthambha Mode B** already issues WireGuard peers keyed to identity —
  the building block (`project_personal_device_join_path`). v1.1 is largely
  *wiring that to discovery*, not inventing a transport.
