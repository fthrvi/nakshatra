# Nakshatra v0.1 — Implementation Plan

**Date:** 2026-05-06
**Status:** Initial plan — to be revised as milestones land. Sibling to:
- [`petals-architecture.md`](petals-architecture.md) — the v0.1 design (the "what")
- [`path-a-vs-path-b-memo.md`](path-a-vs-path-b-memo.md) — C++ feasibility memo (the "why this approach")

This doc is the contract for v0.1. If a v0.1 work item conflicts with this plan, the plan is updated deliberately, not silently.

---

## 1. What v0.1 ships

**The smallest end-to-end Nakshatra cluster.**

- **2 workers**, each holding a contiguous layer range of one model.
- **1 client** that tokenizes a prompt, walks the chain, and samples a token.
- **1 model**, pre-split into per-worker sub-GGUFs by build-time tooling.
- **gRPC over Tailscale** for inter-process transport (no DHT, no libp2p).
- **Patched llama.cpp** linked into each worker as a library; provides partial-model load + per-layer-range decode.
- **Static YAML cluster config** listing each worker's address, port, and assigned layers.

That is it. No rerouting on failure beyond client-side retry. No multi-model. No multi-tenant. No dynamic scheduling. No L3 (NakshatraOS) concepts. Each of those is in scope for a *later* version, not v0.1.

The acceptance test for v0.1 (§7) is the same shape as Phase 0b's success criterion, but with **real partial loading** instead of cheating-and-loading-the-full-model. That single change is the bulk of the work.

## 2. Why this scope

Phase 0a established that llama.cpp's loader rejects partial GGUFs (`experiments/v0.0/partial_gguf_findings.md`). Phase 0b established that the orchestration protocol works once you have a way to intercept hidden state mid-decode (`experiments/v0.0/spike_findings.md`).

What's left is the *actual partial loading* — the patched `llama_decode` that lets a worker hold only its layer slice and run inference end-to-end through that slice. That patch is the work the memo estimated at 10–14 weeks (`path-a-vs-path-b-memo.md` §5.4). v0.1's scope is **exactly that patch plus the minimum scaffolding to demonstrate it works against a real cluster**.

Anything beyond that — production-grade observability, public-network discovery, multi-model registries, dynamic scheduling — adds risk without changing whether the central claim ships.

## 3. Components

### 3.1 Pre-split GGUF tool — Python

**Inputs:** a full GGUF file, a layer-range assignment per worker, output paths.

**Outputs:** N sub-GGUFs, one per worker, where:
- Worker 0 gets `token_embd.weight` + `rope_freqs.weight` + layers `[start_0, end_0)`.
- Worker N-1 gets layers `[start_{N-1}, end_{N-1})` + `output_norm.weight` + `output.weight`.
- All other workers get only their layer-range tensors.
- Every sub-GGUF preserves the original metadata KVs unchanged, including `*.block_count` (the loader patch handles the mismatch — see §3.2).
- A new metadata KV `nakshatra.layer_range = [start, end)` is added so the worker can self-verify on startup.

**Source:** extends [`experiments/v0.0/partial_gguf.py`](../experiments/v0.0/partial_gguf.py). The Phase 0a script is already 90% of this; we just generalize the cut to multiple ranges and add the new metadata KV.

**Estimated effort:** 0.5–1 week. Mostly mechanical.

**Falsifiable check:** for every worker's sub-GGUF, the patched `llama-cli` (§3.2) successfully loads it without error.

### 3.2 Patched llama.cpp — C++ branch

**This is v0.1's load-bearing work.** Two patches against upstream llama.cpp:

#### 3.2.1 Loader patch

Modify `src/llama-model-loader.cpp` (or wherever the model loader currently rejects partial tensor sets) to accept a GGUF whose tensor set is a subset of the architecture's expected set, **provided the new `nakshatra.layer_range` metadata KV is present**. The KV declares which range the file covers; the loader skips tensors outside that range from its expectation list.

A worker without `nakshatra.layer_range` should reject the file with a clear error (we don't want to accidentally accept malformed files).

**Estimated effort:** 1–2 weeks. The loader code is concentrated; the change is "if `nakshatra.layer_range` is set, narrow the expected-tensor set before validation."

**Falsifiable check:** patched `llama-cli` loads a sub-GGUF (per §3.1) without error AND a regular `llama-cli` (unpatched) still rejects it with the same `missing tensor 'blk.X.*'` error we saw in Phase 0a.

#### 3.2.2 Decode patch

Add a new C API entry point:

```c
int llama_decode_layers(
    struct llama_context * ctx,
    const float * hidden_in,        // shape [batch, n_tokens, hidden_size]
    int32_t layer_start,
    int32_t layer_end,
    float * hidden_out              // same shape, owned by caller
);
```

Internally:
- Build the compute graph for layers `[layer_start, layer_end)` only.
- The graph's input tensor is set to `hidden_in` instead of `embd` (the embedding lookup output).
- The graph's output tensor is `l_out-{layer_end-1}` (the residual stream output of the last layer in range).
- Run `llama_decode` machinery on this restricted graph, populating KV cache per-session as usual.

For worker 0 specifically: there's a variant `llama_decode_layers_with_embed(ctx, token_ids, layer_start, layer_end, hidden_out)` that does the embedding lookup internally (since worker 0 receives token IDs, not pre-embedded states).

For the last worker: there's a variant `llama_decode_layers_to_logits(ctx, hidden_in, layer_start, layer_end, logits_out)` that applies output norm + lm_head + produces logits.

**Estimated effort:** 6–10 weeks. The dominant work. Touches `src/llama-graph.cpp`, `src/llama-context.cpp`, and the per-architecture model files (just the Llama family for v0.1).

**Falsifiable check:** running `llama_decode_layers(ctx, full_embed, 0, n_blocks)` on a complete model produces hidden states that, after `output_norm + lm_head + sample`, give the same top-1 token as a standard `llama_decode` on the same prompt. (Sanity check: the patched path on a complete model is equivalent to the unpatched path.)

### 3.3 Nakshatra worker — C++

A daemon that links the patched llama.cpp library and exposes the protocol over gRPC.

**Lifecycle:**
1. Read its sub-GGUF path and layer range from CLI flags.
2. Verify `nakshatra.layer_range` matches the assignment.
3. Load the sub-GGUF via patched loader.
4. Pre-allocate KV cache slice for the assigned layer range.
5. Listen on gRPC port.
6. Serve requests until SIGTERM.

**RPC surface (v0.1):**
- `Info(InfoRequest) → InfoResponse` — capability handshake (§3.4)
- `Forward(ForwardRequest) → ForwardResponse` — single forward pass over the worker's layers (no KV state, used for testing)
- `Inference(stream InferenceStep) → stream InferenceStep` — stateful inference session, one stream per session_id, server-side KV cache

**Estimated effort:** 2–3 weeks (after §3.2 lands). Mostly gRPC plumbing + lifecycle management. The actual compute calls are one-liners against patched llama.cpp.

### 3.4 Nakshatra client — Python

A client library + CLI tool. Python (not Go or C++) because:
- Client is not on the hot path; latency overhead from Python is negligible compared to per-hop network + GPU compute.
- gRPC Python bindings are mature.
- Tokenization libraries (transformers, tiktoken) are Python-first.
- Easier for downstream users (Prithvi, indie devs) to integrate.

**Responsibilities:**
- Tokenize the prompt locally.
- Read cluster config (§3.6).
- Call `Info` on each worker; verify model hashes match across the chain.
- Open `Inference` streams to each worker.
- Walk the chain: send token IDs to worker 0; receive hidden state; forward to worker 1; ...; receive final logits from the last worker; sample.
- Return the next token to the caller.

**Estimated effort:** 1–2 weeks.

### 3.5 Protocol — gRPC `.proto`

```proto
syntax = "proto3";
package nakshatra.v0_1;

service Nakshatra {
  rpc Info(InfoRequest) returns (InfoResponse);
  rpc Forward(ForwardRequest) returns (ForwardResponse);
  rpc Inference(stream InferenceStep) returns (stream InferenceStep);
}

message InfoRequest {}

message InfoResponse {
  string protocol_version = 1;        // e.g. "0.1.0"
  string backend = 2;                 // "llamacpp-cuda" / "llamacpp-vulkan" / etc.
  string model_id = 3;                // e.g. "llama-3.3-70b-q4-k-m"
  bytes  model_content_hash = 4;      // sha256 of the source full GGUF (operator-supplied)
  int32  layer_start = 5;
  int32  layer_end = 6;
  int32  hidden_size = 7;
  string wire_dtype = 8;              // "f16" or "f32"
  int32  kv_cache_tokens_free = 9;
}

message ForwardRequest {
  bytes  hidden_in = 1;               // raw bytes, shape [batch, n_tokens, hidden_size]
  int32  batch = 2;
  int32  n_tokens = 3;
  bool   has_token_ids = 4;           // if true, hidden_in is actually token IDs (for worker 0)
}

message ForwardResponse {
  bytes  hidden_out = 1;
}

message InferenceStep {
  string session_id = 1;
  string step_id = 2;                 // idempotency for retries
  oneof payload {
    bytes hidden_state = 3;           // hidden state going *into* this worker's layer range
    TokenIds token_ids = 4;           // for worker 0: token IDs to embed
    Logits logits = 5;                // for last worker: final logits
  }
  int32  prefix_length = 6;           // tokens already in KV cache
}

message TokenIds { repeated int32 ids = 1; }
message Logits   { bytes raw = 1; }   // shape [vocab_size]
```

**Wire dtype:** v0.1 ships f16. Activation transport is always full-precision (no on-wire quantization), per `petals-architecture.md` §8.3. 16 KB/token at hidden_size=8192.

**Estimated effort:** 0.5 week to define and review. Implementation costs are absorbed into §3.3 and §3.4.

### 3.6 Cluster config — YAML

```yaml
model:
  id: "llama-3.3-70b-q4-k-m"
  source_gguf_sha256: "abcdef..."
  hidden_size: 8192
  num_blocks: 80
  wire_dtype: f16

workers:
  - id: home-pc
    address: "100.101.134.33"      # Tailscale IP
    port: 5500
    layer_range: [0, 40]
    sub_gguf_path: "~/nakshatra/llama-3.3-70b-q4-k-m.0_40.gguf"
  - id: bishwa
    address: "100.109.164.69"
    port: 5500
    layer_range: [40, 80]
    sub_gguf_path: "~/nakshatra/llama-3.3-70b-q4-k-m.40_80.gguf"
```

The client validates the layer ranges form a contiguous `[0, num_blocks)` partition with no gaps or overlaps. Mismatches fail loudly at startup, not at first inference call.

**Estimated effort:** 0.25 week (it's a YAML schema + a validator).

## 4. Build sequence — milestones

The sequence is structured to derisk the biggest unknown (the decode patch) early enough that we can adjust the rest of the plan if it overruns. Milestones are sized in calendar weeks assuming one full-time-equivalent engineer.

| # | Milestone | Effort | Falsifiable check |
|---|---|---|---|
| **M1** | gRPC scaffold | 1 wk | Worker stub starts, `Info` RPC returns mock data; Python client connects and prints the response |
| **M2** | Worker forward, full-model cheating | 2–3 wks | Worker loads full GGUF (no patches yet), exposes `Forward` API. Single-worker chain produces same token as `llama-cli` reference. This is a v0.0-spike-class result, but over gRPC instead of localhost TCP. |
| **M3** | Loader patch | 1–2 wks | Patched `llama-cli` loads a sub-GGUF (from §3.1 tool) for layers `[0, N)` without error. Unpatched `llama-cli` still rejects it. |
| **M4** | Decode patch | 6–10 wks | `llama_decode_layers` on a complete model (whole layer range) reproduces the standard `llama_decode` top-1 token. **This is the dominant milestone; everything else is small in comparison.** |
| **M5** | Two-worker integration | 1 wk | Chain config + two workers each running patched llama.cpp + sub-GGUFs + client walks the chain end-to-end. |
| **M6** | Acceptance test passes | 1 wk | §7's falsifiable criterion: distributed top-1 token matches single-machine reference. |
| **M7** | Operational polish | 1–2 wks | Proper logging, basic metrics, error messages that name the failing worker, documentation for an external operator to spin up a 2-worker cluster from scratch. |

**Total: 13–18 weeks (3.5–4.5 months)** at full-time effort. At half-time, double; at quarter-time (evenings/weekends), triple, per `petals-architecture.md` §9.

**Critical path: M4.** If M4 takes longer than 10 weeks, the v0.1 ship date moves out by exactly that much. M5 onward is small.

**Risk-adjusted plan: aim for M4 = 10 weeks, treat 14+ weeks as the "needs replanning" trigger.** If the patch is fighting upstream churn or hitting unexpected refactors, regroup at the 10-week mark and decide between (a) push through, (b) reduce scope (e.g., drop variant entry points), or (c) defer to v0.2.

## 5. Architecture invariants

These must hold throughout v0.1 implementation. Items 1–6 are restated from `north-star.md` to ensure they stay enforced; items 7–9 are v0.1-specific.

1. Multi-model in the protocol from day one (`ModuleUID` namespacing in v1.0+; v0.1 ships single-model but the protocol envelope must not bake in the assumption).
2. Multi-tenant inference sessions stay multi-tenant.
3. Capability handshake stays rich (§3.5's `InfoResponse` is the v0.1 floor; don't flatten further).
4. Static YAML config is replaceable, not architectural.
5. One worker, multiple models eventually — don't hardcode "the model" as a singleton in the worker process.
6. The protocol does not know about agents.
7. **Patched llama.cpp lives on a tracked branch with monthly upstream rebases.** Skipping more than two cycles is a "stop and catch up" trigger (memo §5.5). Estimated rebase cost: 0.5 day per cycle; 2–5 days for occasional conflicts.
8. **No internal v0.1 fork of llama.cpp's tensor format.** GGUF stays GGUF. Sub-GGUFs are just GGUFs with a metadata KV; they are valid GGUFs that ordinary `llama-cli` could refuse but couldn't corrupt.
9. **The worker process holds no Prithvi state.** It does not know about identities, sessions outside the inference scope, agents, or anything in L3+. It is an inference engine, period (`petals-architecture.md` §0).

## 6. Risks and mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| **M4 overruns 14 weeks** | Medium | High (ship slip) | 10-week regroup checkpoint; reduce scope (variant entry points → defer); commit to monthly upstream rebases so we're not paying back a year of debt at 14 weeks |
| **Upstream llama.cpp refactors `llama_decode` mid-implementation** | Medium | High | Track upstream PRs that touch `src/llama-graph.cpp` / `src/llama-context.cpp`; rebase weekly during M4; have a comprehensive test suite (M2's reference comparison is the floor) that fails loudly on any regression after rebase |
| **gRPC tensor-bytes serialization perf** | Low | Medium | At 16 KB/token across 1 hop, gRPC overhead is sub-millisecond. Instrument latency on M5; if hot, drop to raw protobuf-over-TCP. |
| **KV cache layout changes upstream** | Medium | Medium | KV cache is owned by `llama_context`; v0.1 doesn't manipulate the layout, just allocates per-session. Rebases handle this. |
| **Sub-GGUF format collides with future upstream "shard" format** | Low | Low | Use a clearly Nakshatra-namespaced metadata KV (`nakshatra.layer_range`); if upstream later adds its own partial-load support, we adapt then. |
| **Cross-vendor numerical drift makes acceptance test flaky** | Low (Linux→Linux for v0.1) | Medium | v0.1 only validates same-vendor (e.g., two ROCm Linux machines, or two Vulkan/MoltenVK Macs). Cross-vendor is `petals-architecture.md` §7's v0.3 milestone, not v0.1. |
| **The patched `llama_decode` works but is 10× slower than expected** | Medium | Medium | Instrument tokens/second from M5 onward. If <2 tok/s on hardware that should do 5+, profile; likely cause is cache miss patterns or unnecessary tensor copies in the new entry point. |

## 7. Acceptance criteria

v0.1 ships when **all** of the following hold:

1. Pre-split GGUF tool produces sub-GGUFs that the patched loader accepts and the unpatched loader rejects.
2. Patched `llama_decode_layers` on a complete-model layer range reproduces `llama_decode`'s top-1 token byte-for-byte (numerical equivalence within ε for hidden states is not required; top-1 token equality is).
3. Two-worker cluster, configured per §3.6, with each worker running patched llama.cpp + sub-GGUF, produces the same top-1 next token as a single-machine `llama-cli` running the full model on the same prompt with the same sampling configuration.
4. The single-machine reference is run with greedy decoding (`temperature=0, top_k=1, top_p=1.0`).
5. The cluster runs continuously for at least 10 single-token generations without crashing or producing NaN logits.
6. An external operator can stand up a 2-worker cluster from scratch following only the README + the cluster config schema, in under 1 hour.

Items 1–4 are the architectural ship gate. Items 5–6 are the operational ship gate.

## 8. What's deferred to v0.2+

(Lifted from `petals-architecture.md` §6 with v0.1-specific context.)

- **Server-to-server push (`rpc_push`)** — v0.5. The 1-hop-per-worker latency penalty is acceptable for v0.1's 2-worker test. v0.5 adds it as an opt-in optimization.
- **Dynamic re-routing on worker failure** — v0.5. v0.1 does client-side full-request retry on transient failures (per `petals-architecture.md` §8.2). KV-cache-preserving in-flight recovery is v0.5's contribution.
- **DHT / public-network discovery** — v1.0. v0.1's static YAML is correct for the trust model.
- **Cross-vendor (ROCm + CUDA + Metal in the same chain)** — v0.3 (per architecture §7).
- **Multi-model on one worker** — v0.x; the protocol supports it but v0.1 ships single-model.
- **PEFT / LoRA adapter support** — v1.0+ (architecture §6.4).
- **NRN / payment integration** — Prithvi concern, never Nakshatra's.

## 9. Resolved decisions (originally open questions)

Settled 2026-05-06. Each decision below was picked by weighing the v0.1 trust model + scope + the actual hardware available; if a decision proves wrong during M4 implementation, this doc gets updated, not the code.

- **Sub-GGUF metadata KV names — KEEP `nakshatra.layer_range_*` for v0.1.** M3 already shipped with these names (`nakshatra.layer_range_start`, `nakshatra.layer_range_end`, `nakshatra.has_token_embd`, `nakshatra.has_lm_head`). If upstream llama.cpp later adds its own partial-load API we'll rename then; for v0.1 the names are clearly Nakshatra-namespaced and unlikely to collide.

- **Wire dtype — fp16.** Confirmed per `petals-architecture.md` §8.3. For Llama-3.2-3B (hidden=3072): 6 KB/token; for Llama-3.1-8B (hidden=4096): 8 KB/token; for Llama-3.3-70B (hidden=8192): 16 KB/token. All trivial across one network hop.

- **v0.1 model target — Llama-3.2-3B (the existing `prithvi-q8.gguf`).** Smallest practical Llama-family model that requires splitting to make architectural sense. Already on the home PC, already validated through Phase 0a + Phase 0b + M1 + M2. Iteration speed (model loads in 0.2s, inference in 262ms on CPU) is invaluable during the M4 grind. Llama-3.1-8B and Llama-3.3-70B remain the v0.5+ targets; v0.1's acceptance test does not require a "big" model to validate the architecture.

- **Sampling location — worker-side for v0.1.** Last worker samples (`argmax` for greedy) and returns the int32 token ID over the wire. Saves 128k × 4 = 512 KB per token vs returning full logits. Client-side sampling moves to v0.5 when temperature/top-k/top-p configurability becomes a user-facing requirement. For v0.1's greedy-only acceptance test, worker-side is strictly simpler.

- **Test hardware — home PC + `bishwa`, accepting cross-vendor as a v0.1 risk.** Cross-vendor (Linux/ROCm + Mac/Vulkan-MoltenVK), which the original plan called out as v0.2's milestone. Promoted to v0.1 because waiting on the 4th-PC build for a same-vendor pair adds an unbounded delay. Cross-vendor numerical drift risk is real but mitigated by running both workers in CPU-only mode (`n_gpu_layers=0`) for the v0.1 acceptance test if vendor-specific kernel differences cause top-1 token divergence; we accept the GPU-acceleration claim as a v0.5+ deliverable, not a v0.1 ship gate. **Backup plan:** the same-vendor pair (`bishwa` + `mentorings-imac-pro`, both Vulkan-MoltenVK) is available if cross-vendor proves intractable.

## 10. How this doc is used

- When starting any v0.1 work item, locate it in §3 (component) and §4 (milestone). If it's not there, either (a) the plan is wrong and needs an update, or (b) the work isn't on the v0.1 critical path.
- When a v0.1 work item conflicts with an invariant in §5, the invariant wins. Update the plan to redesign around the invariant.
- When M4 hits a snag, return to §6 (risks) before reaching for new code; the mitigation may already be specified.
- When v0.1 ships, this doc becomes a historical artifact. The successor is `docs/v0.5-implementation-plan.md` or similar.
